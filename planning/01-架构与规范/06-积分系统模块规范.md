## 1. 文档目的

先把积分体系的产品口径、计价规则、模块边界和代码分工定清楚，再进入代码实现。

本文件作为后续 `22` 执行代码的唯一范围依据。没有写入本文件的功能，不在下一轮代码实现范围内。

## 2. 当前结论

- 积分必须作为独立模块，不散落在生成、充值、历史记录和 UI 代码里。
- 每张照片的积分消耗由统一定价函数计算。
- 生成前显示预计消耗，生成成功后按实际成功图片数扣费。
- 生成失败、接口超时、上游无图返回不扣积分。
- 充值、扣费、退款都进入同一个积分明细。
- 本阶段先做本地模拟充值和本机账本，不接真实支付。

## 5. 目标

- 定义每张照片消耗多少积分。
- 定义充值积分和生成扣费的业务规则。

## 6. 非目标

- 不接微信、支付宝、Stripe 或其他真实支付。
- 不做用户登录、手机号登录或企业账号体系。
- 不做后台运营系统。
- 不改模板库、提示词库和图片生成模型配置。
- 不把积分规则写死在 UI 文案里。
- 不把图片生成失败记录当作扣费成功。

## 7. 积分单位

### 7.1 基本口径

- `积分` 是站内消耗单位。
- 上传参考图不消耗积分。
- 浏览模板、搜索模板、填写 Prompt 不消耗积分。
- 只有请求生成并成功返回图片时才扣积分。
- 多张图按实际成功图片张数计费。

### 7.2 扣费时机

推荐流程：

1. 生成前：服务端计算预计消耗。
2. 生成前：可用积分不足则拒绝提交。
3. 生成中：不提前扣费。
4. 生成成功：按实际返回图片数扣费。
5. 生成失败：不扣费。
6. 已扣费但图片保存失败：记录异常，支持人工退款。

说明：先不做预冻结，避免本地账本复杂化。后续接真实支付和并发队列时，再增加冻结/释放机制。

## 8. 每张照片积分规则

### 8.1 基础价格

| 生成类型 | 判断条件 | 基础积分 |
| --- | --- | ---: |
| 普通生成 | 无参考图，文生图 | 20 / 张 |
| 图片编辑 | 有参考图，图生图或改图 | 30 / 张 |
| 身份确认基准 | 身份确认用参考照 | 60 / 组 |
| 模板样片 | 场景样片、小批试风格 | 20 / 张 |
| 批量成片 | 正式交付图片 | 20 / 张 |

### 8.2 加价规则

| 条件 | 加价 |
| --- | ---: |
| `quality = high` | +10 / 张 |
| 最长边大于等于 2048 | +20 / 张 |
| 最长边大于等于 3840 | +60 / 张 |
| 有 2 张及以上参考图 | +10 / 张 |

### 8.3 计算方式

```text
单张积分 = 基础价格 + 质量加价 + 尺寸加价 + 多参考图加价
总积分 = 单张积分 * 请求张数
最终扣费 = 单张积分 * 实际成功图片张数
```

### 8.4 示例

| 场景 | 参数 | 预计积分 |
| --- | --- | ---: |
| 普通生成 1 张 | auto / auto / 1 张 | 20 |
| 普通生成 4 张 | auto / auto / 4 张 | 80 |
| 图片编辑 1 张 | 1 张参考图 / auto | 30 |
| 图片编辑 2 张 | 1 张参考图 / auto / 2 张 | 60 |
| 高质量普通图 1 张 | quality high | 30 |
| 2K 高质量图 1 张 | 2048 边长 / high | 50 |
| 4K 高质量图 1 张 | 3840 边长 / high | 90 |
| 身份确认基准 1 组 | front / side / back | 60 |

## 9. 可用积分不足规则

- 可用积分小于预计消耗时，生成按钮不可提交。
- 页面显示：`可用积分不足，还差 X 积分`。
- 充值按钮跳转到我的积分页。
- 不允许前端绕过可用积分检查；服务端仍必须二次校验。

## 10. 积分明细

### 10.1 流水类型

| 类型 | 含义 |
| --- | --- |
| `recharge` | 充值入账 |
| `spend` | 生成扣费 |
| `refund` | 退款返还 |
| `adjustment` | 人工调整 |

### 10.2 字段

| 字段 | 说明 |
| --- | --- |
| `id` | 流水 ID |
| `type` | 流水类型 |
| `credits` | 积分变化，入账为正，扣费为负 |
| `balanceAfter` | 变动后可用积分 |
| `title` | 客户可读标题 |
| `reason` | 内部原因 |
| `taskId` | 关联生成任务 |
| `packageId` | 关联充值套餐 |
| `imageCount` | 实际成功图片数 |
| `unitCost` | 单张积分 |
| `createdAt` | 创建时间 |

## 12. 模块分区

### 12.1 服务端模块

建议新增：

```text
server/credits/config.mjs

```

职责：

| 文件 | 职责 |
| --- | --- |

## 13. API 边界

### 13.1 查询积分

```text
GET /api/credits
```

返回：

```json
{
  "ok": true,
  "balance": 330,
  "ledger": [],
  "packages": []
}
```

### 13.2 预估生成消耗

```text
POST /api/credits/estimate
```

入参：

```json
{
  "params": {
    "size": "1024x1024",
    "quality": "auto",
    "count": 1
  },
  "references": []
}
```

返回：

```json
{
  "ok": true,
  "unitCost": 20,
  "estimatedCost": 20,
  "balance": 330,
  "enough": true
}
```

### 13.3 买积分

```text
POST /api/credits/recharge
```

入参：

规则：

- 买积分前必须先登录。
- 未登录时返回 `请先登录，再买积分`。
- 客户前台按钮只写 `先登录` 或 `买积分`，不写内部接口词。

```json
{
  "packageId": "starter"
}
```

### 13.4 生成扣费

`POST /api/generate` 内部调用积分服务：

1. 计算预计消耗。
2. 校验可用积分。
3. 生成图片。
4. 按实际成功图片数扣费。
5. 返回扣费结果。

返回补充字段：

```json
{
  "creditCost": 20,
  "creditBalance": 310,
  "creditLedgerId": "credit_xxx"
}
```

## 14. UI 分区

### 14.1 顶部

只展示：

- 当前可用积分
- 我的积分入口
- 注册开通入口

不展示：

- 长说明
- 套餐列表
- 完整流水
- 连接设置
- 测试连接

### 14.2 生成功能区

在生成按钮附近增加一个紧凑条：

```text
预计消耗 20 积分 · 当前可用积分 330
```

可用积分不足时：

```text
可用积分不足，还差 20 积分
```

### 14.3 积分功能区

分三块：

1. 可用积分概览
2. 买积分套餐
3. 积分明细

界面要求：

- 保持一屏内可扫读。
- 充值套餐最多 4 个首屏展示。
- 积分明细默认展示最近 8 条。
- 历史完整记录后续再做“更多记录”。

## 15. 生成历史与积分关系

| `creditUnitCost` | 单张积分 |

## 17. 执行顺序

1. 抽服务端积分模块。
2. 补定价单元测试。
3. 接 `/api/credits/estimate`。
4. 接充值和流水。
5. 接 `/api/generate` 扣费。
6. 接前端生成页预计消耗。
7. 接可用积分不足状态。
8. 做视觉检查。
9. 重载 9999 常驻服务。
10. 写 build 报告。

## 18. 验证方案

文档完成后，代码阶段必须验证：

- `npm run lint:strict`

- `npm run typecheck`
- `npm run verify:visual`

新增测试必须覆盖：

- 可用积分不足拒绝生成。
- 生成失败不扣费。
- 部分成功按实际图片数扣费。
- 充值入账生成流水。
- 不同客户端账本隔离。
