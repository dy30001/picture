## 2. 现在是不是一套系统

结论要分两层写：

### 2.1 运行态

是。

- `package.json` 当前默认启动只有 `server/index.mjs --port 9999`
- `curl http://127.0.0.1:9999/api/health` 已返回健康结果
- `server/index.mjs` 已直接读取 `final_4k/` 作为摄影棚成片预览目录

### 2.2 目录治理态

还没有完全收口。

- 源码仍主要放在 `public/`、`server/`、`scripts/`、`tests/`
- `generated/`、`review/`、`final_4k/`、`reference/` 仍是业务产物目录
- `成片/` 还在仓库里，属于待后续清理的历史重复目录

所以现在最准确的话是：

- 当前已经是一套主应用在跑
- 但目录还要继续整理成“一个代码根目录 + 子文件夹”

## 3. 每个板块怎么开发

### 3.1 统一入口与部署

目标：

- 外部只认 `9999`
- 所有正式文档、启动命令、Nginx 反代都写同一口径

开发要求：

1. `package.json`、`README.md`、部署文档保持一致
2. 不再新增第二个客户可见端口
3. 如果出现历史调试入口，只能标注为兼容或临时用途

### 3.2 主应用摄影棚

目标：

- 客户第一眼看到摄影棚流程，不是技术工具集合

开发要求：

1. 首屏只保留场景、身份、样片、成片摘要
2. 长模板库、设置项、历史记录默认折叠或次级入口
3. 选片与交付入口作为主应用里的子流程

### 3.3 生图与模板能力

目标：

- 继续保留专业生图能力，但不抢主流程叙事

开发要求：

1. 模板、生成、历史走统一 API
2. 模板列表轻量加载，详情按需取
3. 服务端负责历史记录和失败保护

### 3.4 样片管理

目标：

- 把样片管理从“临时看图”升级成正式模块

开发要求：

1. 统一样片对象、状态、操作记录
2. 样片与成片分开，不混列表
3. 离线 repair queue 要能映射回正式状态

### 3.5 复核与交付

目标：

- 成片复核和交付继续保留，但只认主应用口径

开发要求：

1. 正式交付只认 `final_4k/`
2. 交付页默认不混入样片、失败图、替换前旧图
3. `成片/` 只留历史兼容，不再新增同步依赖

### 3.6 身份资产与场景生产

目标：

- 固定 `三视图 -> 样片 -> 批量场景 -> 复核 -> final_4k`

开发要求：

1. 每个场景包保留计划、脚本、接触表、报告
2. 执行报告 必须能追到输出目录
3. 模板样片批次不直接算正式交付

### 3.7 账号、积分、支付

目标：

- 账号和权益全部归主应用

开发要求：

1. 不再拆第二套账号态
2. 积分规则只以服务端为准
3. 支付订单和扣费校验走统一接口

### 3.8 视觉资产与质量门禁

目标：

- 页面紧凑、清楚、可扫读

开发要求：

1. 历史和明细默认折叠
2. 不做松散大留白页面
3. 每次改动都带最小验证闭环

## 4. 当前目录职责

| 目录 | 当前职责 |
| --- | --- |
| `public/` | 主应用前端页面、状态渲染、视觉资产入口 |
| `server/` | 主应用服务、API、静态资源挂载、账号/积分/支付能力 |
| `scripts/` | 启动、构建、视觉检查、批量生成脚本 |
| `src/` | 可测试的核心逻辑 |
| `tests/` | Node 服务和逻辑测试 |
| `generated/` | 生成过程源图与中间 4K 结果 |
| `review/` | 接触表、截图、repair queue |
| `final_4k/` | 唯一正式成片目录 |
| `成片/` | 历史重复目录，待清理 |
| `planning/` | 规格、计划、执行报告 |
| `reference/` | 参考图与基准素材 |

## 5. 完成标准

文档和代码同时满足下面 6 条，才算这轮统一到位：

1. 默认启动和健康检查都只认 `9999`
2. 所有正式文档把 `final_4k/` 写成唯一正式成片目录
3. `成片/` 不再被脚本和文档当成并行正式输出
4. 样片管理、复核交付在文档里都有明确板块归口
5. 目录职责清楚，不再把历史拆分态写成现行结构
6. 核心检查至少包含 lint、test、typecheck 和变更脚本自检
