# 墨境前端设计规范

日期： 2026-05-02
模式： `doc`
工作区： `/Users/dy3000/code/pic`

## 0. 规范定位

这份规范定义墨境前端界面的长期设计口径，用于后续新增页面、重构模块、检查 UI 质量和做可视化验收。

适用范围：

- `public/index.html`
- `public/styles.css`
- `public/app.js`
- 客户前台、图片创作、我的作品、注册开通、我的
- 后续所有同一主应用下的新页面和弹窗

不适用范围：

- 不定义品牌 logo 的完整使用规则，logo 规则见 `07-品牌视觉设计规范.md`
- 不定义图片生成提示词规范，提示词规则见 `03-流程与提示词/05-提示词系统规则.md`
- 不把墨境做成营销型官网或普通 SaaS 落地页

## 1. 前端产品定位

墨境前端是客户影像交付工作台，不是模板展示页，也不是参数控制台。

标准定位：

```text
客户先看当前状态和下一步，再看样片证据，最后进入生成与交付。
```

设计目标：

- 客户一眼知道现在做到哪一步。
- 客户只看到当前需要做的动作。
- 交付人员能快速检查身份、场景、样片、积分和生成记录。
- 技术设置、接口调试、历史细节默认后置。

## 2. 信息架构

### 2.1 一级菜单

当前设计采用 5 个一级业务板块：

| 顺序 | 菜单 | 用途 | 主动作 |
| --- | --- | --- | --- |
| 1 | `拍摄定制` | 客户项目主流程 | 开始选套餐 / 开拍 |
| 2 | `图片创作` | 模板库与智能生图 | 使用模板 / 生成图片 |
| 3 | `我的作品` | 作品、删除恢复和下载 | 查看 / 恢复 / 下载 |
| 4 | `注册开通` | 注册、登录和账号归属 | 注册 / 登录 |
| 5 | `我的积分` | 积分、充值、订单和流水 | 购买积分 / 刷新 |

规则：

- 不把 `模板库`、`智能生图`、`连接设置` 放成一级菜单。
- 不使用旧口径 `账户权益`，客户前台统一写 `我的积分`。
- 一级菜单文案保持短句，不超过 4 个中文字符。
- 一级菜单是业务逻辑里的 5 个板块，不是界面排版规则。
- 当前菜单必须有明确 active 状态。
- 前端可以按响应式设计横排、换行或折叠，布局方式不改变业务板块定义。
- 后续如果业务设计要调整板块数量、命名、顺序或归属，先改文档确认，再改代码。

### 2.2 二级菜单

二级菜单只承载当前一级菜单内的锚点或视图切换。

当前约定：

- `拍摄定制`：样片集、生图、身份确认、选模板、生成
- `图片创作`：模板库、智能生图
- `我的作品`：全部作品、已删除
- `注册开通`：注册、登录
- `我的积分`：我的积分、充值中心、充值订单、积分流水

补充口径：

- `图片创作` 当前实现仍走 `模板库`、`智能生图` 两条次级路由。
- 客户前台主区推荐先收口为 `趣味生图`、`套图写真` 两个区块，避免把页面做回模板池。

规则：

- 客户界面不使用 `账户权益`、`余额`、`三视图`、`数字底片`、`加料项`、`继续` 这类内部词。
- 一律写成 `我的作品`、`我的积分`、`可用积分`、`身份确认`、`参考照`、`可选功能`、`主动作`。
- 二级菜单必须随一级菜单切换。
- 二级菜单不展示技术入口。
- 二级菜单超过一行时优先压缩间距，不扩大页面顶部高度。

## 3. 页面主流程

### 3.1 拍摄定制首页

拍摄定制是默认首屏，顺序固定：

1. 当前项目摘要
2. 当前工作流状态
3. 下一步主动作
4. 场景套餐
5. 右侧当前订单
6. 样片方向和成片交付

硬规则：

- 首页不展示完整模板库。
- 首页不展示长 prompt。
- 首页不展示接口设置和调试日志。
- 当前状态、当前证据、下一步动作必须在首屏内可见。

### 3.2 客户工作流

前端流程固定为：

```text
选择场景 -> 上传参考照 -> 确认像本人 -> 选择样片方向 -> 批量成片 -> 复核与交付
```

每一步必须有：

- 当前状态
- 是否已完成
- 下一步动作
- 必要证据

状态文案要面向客户：

- 用 `待上传`、`已确认`、`待选择`、`可生成`
- 不用 `pending`、`done`、`stage_3` 等内部字段

## 4. 布局规范

### 4.1 桌面端

推荐宽度：

- 页面外壳：`min(1500px, calc(100vw - 36px))`
- 主工作区：左侧主内容 + 右侧当前订单
- 右侧栏宽度：约 `360px - 420px`

桌面端规则：

- 右侧栏可 sticky，但只放当前任务，不堆历史。
- 主区域卡片尽量两列或三列，避免单列拉太宽。
- 同级卡片高度不强制完全一致，但主信息位置要稳定。
- 首屏不要做大面积纯装饰 Hero。

### 4.2 移动端

移动端顺序：

1. 顶部品牌与主菜单
2. 二级菜单
3. 当前状态
4. 下一步主动作
5. 当前证据
6. 详情和历史

移动端规则：

- 右侧栏必须下沉为普通内容流。
- 按钮高度不低于 `42px`。
- 图片卡片必须保留稳定比例。
- 任何控件不得依赖 hover 才能完成核心操作。

### 4.3 留白与密度

墨境前端要“紧凑但不拥挤”。

建议：

- 页面区块间距：`14px - 20px`
- 卡片内边距：`12px - 18px`
- 小型控件间距：`6px - 10px`
- 主要按钮高度：`44px - 48px`

避免：

- 大段空白 Hero
- 大标题占满半屏
- 卡片套卡片
- 重复状态在多个地方同时出现

## 5. 视觉系统落地

### 5.1 色彩使用

前端色彩必须继承品牌规范。

当前 CSS 变量基线：

| 变量 | 用途 |
| --- | --- |
| `--paper` | 页面主背景 |
| `--paper-soft` | 页面底色和弱分区 |
| `--paper-strong` | 卡片和表单底 |
| `--ink` | 主标题和正文 |
| `--ink-soft` | 次级标题 |
| `--muted` | 说明文案 |
| `--line` | 边框和分割线 |
| `--line-strong` | 强边框和金线 |
| `--cinnabar` | 主动作和关键状态 |
| `--jade` | 次强调和已完成状态 |
| `--gold` | 细节、边线、装饰 |

规则：

- 一屏只能有一个红色主动作。
- 青色用于辅助状态，不和主按钮抢层级。
- 金色只做线和细节，不做大面积按钮底。
- 背景不使用纯白铺满，优先使用宣纸暖底。

### 5.2 字体与字号

字体：

- 品牌标题：`Songti SC`、`STSong` 或同类宋体
- 界面正文：`PingFang SC`、`Hiragino Sans GB`、`Microsoft YaHei`

字号建议：

| 场景 | 建议 |
| --- | --- |
| 顶层页面标题 | `26px - 32px` |
| 区块标题 | `18px - 22px` |
| 卡片标题 | `15px - 18px` |
| 正文说明 | `13px - 15px` |
| 标签、表头、辅助信息 | `11px - 13px` |

规则：

- 不用 viewport 宽度缩放字体。
- 字距保持 `0` 或轻微正字距，不用负字距。
- 容器内标题不使用 Hero 级字号。

### 5.3 圆角与阴影

控件圆角：

- 按钮、输入框、标签：`8px`
- 卡片：`8px - 14px`
- 品牌图标或大预览：可更大，但要有明确理由

阴影：

- 工作台卡片使用轻阴影。
- hover 阴影只做轻微增强。
- 不使用强发光和玻璃拟态泛滥。

## 6. 组件规范

### 6.1 顶部导航

顶部导航承担全局方向，不承担说明功能。

必须包含：

- 品牌图标
- 当前产品标题
- 一级菜单
- 我的积分快速状态

不得包含：

- API URL
- API Key
- 模型列表
- 调试状态

### 6.2 状态条

状态条用于告诉用户“当前下一步”。

标准结构：

```text
当前下一步 / 当前阶段 / 当前场景 / 当前样片 / 当前积分 / 下一步按钮
```

规则：

- 状态条文案必须短。
- 按钮只保留一个主动作。
- 当前步骤改变时，状态条是状态单一来源。

### 6.3 场景套餐卡

场景套餐卡用于选择客户目标，不用于展示完整模板细节。

必须包含：

- 场景名称
- 一句话说明
- 代表图
- 关键标签
- 价格或积分信息，如果当前业务需要

规则：

- 以 `婚纱照`、`情侣照`、`闺蜜照`、`儿童10岁照`、`女生写真`、`夕阳红` 为一级业务语言。
- 选中状态必须明显。
- 不能把 prompt 文本放进卡片。

### 6.4 样片预览

样片预览用于降低客户决策成本。

必须包含：

- 当前场景
- 当前样片方向
- 大图预览
- 缩略图切换
- 确认样片方向动作

规则：

- 图片区域必须有固定比例，加载中不能导致布局跳动。
- 预览弹窗必须可关闭。
- 样片确认后才能进入批量成片。

### 6.5 表单

表单用于注册、登录、生成设置和必要输入。

规则：

- label 必须明确。
- placeholder 只做示例，不替代 label。
- 错误提示放在字段附近或表单动作附近。
- 注册和登录切换使用分段控件或 tab，不堆两套表单。
- 密码、验证码、邮箱字段遵守浏览器 autocomplete 语义。

### 6.6 弹窗

弹窗只用于当前操作的补充确认。

允许弹窗：

- 样片大图预览
- 图片尺寸设置
- 订单支付结果
- 删除确认

不允许弹窗：

- 长篇说明
- 完整模板库
- 常驻工作流状态

### 6.7 智能生图主次结构

`图片创作 -> 智能生图` 不是参数控制台，而是客户直接开拍页。

固定主次：

- 主体：`上传照片`、`画面描述`、`开拍`
- 辅助：`可选功能`、`尺寸`、`质量`、`格式`、`数量`、`模板库回跳`、`充值`

规则：

- 桌面端首屏必须同时看到上传照片入口和开拍主动作。
- 上传照片和开拍必须留在主列，不能被参数卡、提示词补充区或工具按钮压到次级位置。
- 可选功能本质上是“在主描述上追加效果”，不能表现成独立主流程。
- 提示词“怎么写”的说明、写法参考、模板借句方式，默认放在侧边辅助区，不占主区首屏。
- 基础参数、尺寸和模板回跳收进辅助区，不和主区并排争抢视觉中心。
- 文本输入区不允许成为唯一最大块；如果页面显得先看到大文本框、后看到上传和开拍，说明主次错误。
- 移动端顺序固定为：当前状态 -> 上传照片 -> 画面描述 -> 开拍 -> 辅助设置。

## 7. 交互规范

### 7.1 主动作

每个页面或区块只允许一个视觉主动作。

示例：

- 拍摄定制：`下一步`
- 场景套餐：`选择套餐`
- 样片方向：`确认样片方向`
- 成片交付：`开拍成片`
- 我的积分：`立即充值`

### 7.2 动效

动效只服务状态变化，不做装饰表演。

建议：

- hover：`120ms - 180ms`
- 页面状态切换：`180ms - 280ms`
- 弹窗进入：`180ms - 240ms`

规则：

- 动效不能影响可读性。
- 移动端不依赖 hover。
- 避免大面积漂浮和闪烁。

### 7.3 键盘与焦点

要求：

- 按钮和输入框必须可键盘访问。
- 弹窗打开后焦点进入弹窗。
- 弹窗关闭后焦点回到触发元素。
- `Enter` 提交表单前必须保证不会误触高成本操作。

## 8. 文案规范

### 8.1 UI 文案原则

- 一句话说结果。
- 少写解释，多给动作。
- 技术词后置。
- 不把内部错误直接丢给客户。

推荐：

- `上传参考照`
- `确认像本人`
- `看样片`
- `开拍成片`
- `充值中心`
- `积分已刷新`

避免：

- `开始执行 pipeline`
- `调用接口失败，请检查 Base URL`
- `模型参数缺失`
- `stage status pending`

### 8.2 状态文案

状态词固定：

| 类型 | 可用文案 |
| --- | --- |
| 等待 | 待上传、待选择、待确认 |
| 进行 | 生成中、读取中、处理中 |
| 完成 | 已确认、已生成、已到账 |
| 风险 | 需复核、积分不足、上传失败 |

## 9. 可访问性与兼容

### 9.1 基线要求

- 正文对比度保持可读。
- 关键按钮不能只靠颜色表达状态。
- 图片必须有必要的 `alt` 或使用空 alt 隐藏装饰图。
- 表单必须有 label。
- 弹窗必须有明确标题。

### 9.2 目标设备

至少验证：

- 桌面：`1440 x 1000`
- 移动：`390 x 844`

### 9.3 溢出控制

硬规则：

- 页面不能出现横向滚动。
- 按钮文字不能溢出。
- 表格和长列表必须进入内部横向滚动容器。
- 图片预览不能撑破视口。

## 10. 验证标准

### 10.1 自动检查

优先使用仓库内脚本：

```bash
npm run lint:strict
npm test
npm run typecheck
npm run verify:visual
```

如果只是文档任务，不强制跑完整前端检查，但必须核对引用路径和文档索引。

### 10.2 UI 视觉检查

UI 任务必须额外检查：

- 桌面截图
- 移动端截图
- 首屏是否能看到当前状态和下一步
- 是否有过宽、过散、重复、溢出
- 一级菜单和二级菜单是否符合规则
- 技术入口是否从客户主流程中隐藏

### 10.3 当前可复用的验收脚本

当前视觉验收脚本：

```text
scripts/visual-check.mjs
```

它已经覆盖：

- 5 个一级菜单顺序
- 二级菜单分组
- 注册 / 登录流程入口
- 我的积分入口
- 场景套餐和样片流程
- 模板库到智能生图路径
- 桌面和移动端截图
- 横向溢出数量阈值

## 11. 新页面检查清单

新增页面或大改 UI 前，必须过下面 12 项：

1. 是否属于 5 个一级菜单之一。
2. 首屏是否有当前状态和下一步。
3. 是否只有一个视觉主动作。
4. 是否复用现有 CSS 变量。
5. 是否复用现有按钮、卡片、状态、表单模式。
6. 是否没有卡片套卡片。
7. 是否没有长说明占据主流程。
8. 移动端是否没有横向滚动。
9. 图像是否有稳定比例。
10. 表单是否有 label 和错误提示。
11. 技术入口是否后置。
12. 是否能被 `scripts/visual-check.mjs` 或对应截图验证。

## 12. 当前前端硬规则

- 默认首屏是 `拍摄定制`。
- 客户前台不展示 `连接设置`、`测试连接`。
- `模板库` 和 `智能生图` 属于 `图片创作`。
- `图片创作` 的客户前台主区优先写 `趣味生图`、`套图写真`，不是并排再加第三个模板区。
- `智能生图` 必须保持“上传照片 / 开拍”为主，“可选功能 / 参数”为辅。
- 历史内容默认不抢主流程。
- 人像流程坚持 `先确认像本人，再出模板样片`。
- 主界面只保留当前结果、当前进度、阻塞点、下一步和证据。
- 页面如果显得太散，必须优先收紧，而不是再加装饰。
