# 客户选片与部署门禁规格 v01

Date: 2026-05-02
Mode: doc
Workspace: `/Users/dy3000/code/pic`

## 0. 当前总口径

1. 客户统一访问 `http://<host>:9999`
2. 当前正式运行服务是 `server/index.mjs`
3. 正式成片目录是 `final_4k/`
4. 样片和成片必须分开
5. `成片/` 只保留历史兼容数据，不再作为正式交付目录

## 1. 客户主流程

客户不需要理解模型、模板库或内部文件夹，只需要按步骤完成选择、确认和交付。

主流程固定为：

1. 选择场景
2. 上传参考照
3. 确认三视图/身份基准
4. 查看模板样片
5. 确认后批量生成
6. 复核正式成片
7. 下载或确认交付

## 2. 页面要求

### 2.1 首页

首页只回答三个问题：

- 现在做什么场景
- 当前流程到哪一步
- 下一步点哪里

推荐布局：

1. 顶部：产品名、当前项目、服务状态
2. 主流程条：场景、身份、样片、成片、交付
3. 当前步骤面板：只展示本步骤需要的操作
4. 证据区：默认折叠，展示健康检查、生成记录、接口状态

### 2.2 身份确认页

必须做到：

- 上传参考图
- 展示三视图预览
- 提供确认/返修动作
- 未确认前不开放正式批量成片主动作

### 2.3 样片页

样片页用于选风格，不承担最终交付。

必须做到：

- 推荐模板优先
- 长模板库默认折叠
- 样片区和正式成片区分开

### 2.4 成片与交付页

交付页只展示正式可用结果。

必须做到：

- 正式图片来自 `final_4k/`
- 样片、失败图、被替换旧图默认不进交付页
- 交付页只保留一个主动作：下载或确认交付

## 3. 状态门禁

### 3.1 身份门禁

只有在以下状态下才允许进入样片或成片下一步：

- `待上传`
- `待三视图`
- `待确认`
- `已确认`
- `需返修`

硬规则：

- 未确认三视图前，不能启动批量成片

### 3.2 成片门禁

批量成片只在身份和样片都通过后启动。

状态建议：

- `排队中`
- `生成中`
- `待复核`
- `需返修`
- `可交付`

## 4. 部署门禁

### 4.1 启动前检查

- 工作区：`/Users/dy3000/code/pic`
- 依赖完整：`npm install` 已完成或 `node_modules` 存在
- 当前正式端口：`9999`
- 当前正式服务：`server/index.mjs`
- 当前正式成片目录：`final_4k/`

### 4.2 启动命令

```bash
node server/index.mjs --host 0.0.0.0 --port 9999
```

说明：

- 这是当前唯一正式启动命令
- 旧的 `identity-workflow` 独立启动方式不再作为现行部署口径

### 4.3 健康检查

```bash
curl -fsS http://127.0.0.1:9999/api/health
curl -fsS http://SERVER_IP:9999/api/health
```

通过标准：

- 返回 `ok: true`
- 服务名正确
- 页面可正常加载主界面

### 4.4 页面检查

部署前至少检查：

1. 首页能打开
2. 场景、身份、样片、成片主流程可见
3. 样片区和成片区没有混在一起
4. 没有页面级横向滚动
5. 按钮、标题、卡片不松散

## 5. 目录与交付门禁

### 5.1 正式交付目录

正式交付只认：

```text
final_4k/
```

### 5.2 历史兼容目录

历史重复目录是：

```text
成片/
```

它当前只允许：

- 保留历史数据
- 作为清理前的兼容痕迹

它当前不允许：

- 继续作为正式交付目录
- 继续作为脚本默认同步目标
- 继续作为正式文档里的交付口径

1. 客户只记住 `9999`

3. 样片和成片不混口径
4. 正式成片只认 `final_4k/`
