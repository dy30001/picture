## 1. 产品定位

墨境下一阶段的定位应从：

```text
GPT Image 2 本地图像生成工作台
```

升级为：

```text
个人AI摄影棚：客户提交照片和影像方向，后台制作并上传成果，客户只查看最终成片。
```

核心判断：

- 竞品强在“用户感觉自己拥有了一个随叫随到的摄影棚”。
- 墨境不能只展示 2300+ prompt 模板，否则用户会觉得自己还在操作工具。
- 墨境前台应拆成面向注册用户的客户平台，后台另建面向管理员和工作人员的制作平台。
- 客户平台只做：样片集、提交资料、等待制作、查看成片。
- 后台制作平台负责：接收订单、审核资料、制作成片、上传成果、通知客户、完成交付。
- 墨境的差异化不在“比别人模板更多”，而在“半专业交付流程”：客户少操作、后台可追踪、成果可复核、最终可交付。

## 1.0 当前冻结口径

截至 2026-05-05，本文档作为“个人AI摄影棚”的主设计文档，当前冻结口径如下：

1. 产品不是普通生图工具，而是客户影像交付工作台。
2. 客户前台只保留 6 个业务板块：`首页`、`影像定制`、`图片创作`、`我的作品`、`登录`、`我的积分`。
3. `影像定制` 的客户主线只保留：`样片集 -> 提交资料 -> 制作状态 -> 查看成片`。
4. 后台制作平台不属于客户前台业务板块，后续通过独立内网端口和管理员登录承载。
5. 宣传主张统一为：`让向往的生活，先在照片里发生`。
6. 客户界面只展示结果、当前状态、阻塞点、下一步和必要证据。
7. 内部生产词、模型参数、长 prompt、测试连接和后台上传动作不进入客户前台。
8. 统一实现入口是 `server/index.mjs --port 9999`，正式成片目录只认 `final_4k/`。

如果后续业务设计要改菜单数量、命名、顺序、后台入口或客户状态，先修改本文档和相关索引，再进入代码修改。

## 1.1 当前产品结构更新

用户最新确认的产品结构：

```text
平台 A：客户平台
- 面向注册用户。
- 用于看样片、提交资料、等待制作和查看成片。
- 客户只上传照片、选择影像组合、勾选风格或模板方向、查看结果。
- 客户平台不展示 `三视图`、`人物确认图`、`像不像审核`、`后台上传` 等内部生产动作。

平台 B：后台制作平台
- 面向管理员和工作人员。
- 用于查看订单、审核资料、制作成片、上传成果、通知客户和完成交付。
- 后台可以保留制作记录、复核记录、返修记录和成果版本。
- 后台入口必须管理员登录，和客户账号体系分开。
```

界面口径：

- 客户平台只展示样片、资料提交、制作状态和成片结果。
- 后台制作平台可以展示订单、客户资料、内部状态、成果上传和通知动作。
- 模板是客户选择影像方向的输入，不再让客户承担正式生成动作。

## 2. 竞品模式学习

### 2.1 Photo AI

公开口径：

- 定位为 AI Photographer。
- 用户上传自拍，创建自己的 AI model。
- 创建 AI model 后，可以在任何姿势、地点或动作中拍照。
- 支持 100s of photo packs。
- 支持多人、视频、换装、产品视频。
- 页面强调训练一次后可快速生成无限照片。

可借鉴：

- 用“摄影师 / 摄影棚”表达产品，而不是“生成器”。
- 把身份建模当作资产，而不是一次性上传。
- 用 photo packs 降低 prompt 门槛。
- 多人、视频、换装可以作为后续增长功能。

不直接照搬：

- 他们更偏自助 SaaS；墨境更适合做成“服务交付工作台 + 客户选片台”。
- 当前阶段不做完整模型训练，优先沿用参考照输入、后台资料审核和人工制作门禁。

### 2.2 pic.mx

公开口径：

- 定位为 AI Photo Studio 和 personal AI photographer。
- 用户上传自拍，获得个人 AI model。
- 强调任意风格、任意地点、秒级出图。
- 支持情侣、朋友、家庭多人模型合照。
- 强调隐私：模型只属于用户，可删除。
- 对比传统摄影棚：成本低、速度快、不需要预约。

可借鉴：

- “个人AI摄影棚”是最适合墨境的前台表达。
- 多人模型组合非常适合墨境的情侣照、闺蜜照、夕阳红和家庭照。
- 隐私承诺必须前置，尤其是真人参考照上传场景。
- 和传统摄影棚做对比时，重点说省预约、省拍摄、省选景、省返工。

### 2.3 PhotosAI

公开口径：

- 用户上传 10-20 张自拍。
- 训练个人 AI model 约 5 分钟。
- 按 LinkedIn、Instagram、Tinder、Travel、Fashion 等 photo packs 使用。
- 强调高相似度、真实皮肤、高分辨率、手部细节、数据安全。
- 订阅套餐按照片数、模型数、清晰度、优先生成等分级。

可借鉴：

- 上传环节必须给明确数量和质量标准。
- 场景包要按真实用途命名，不按内部风格命名。
- 质量卖点不能只说“高清”，还要说“像本人、皮肤真实、手部自然、可打印”。
- 套餐分级可以围绕身份数、场景数、成片数、高清下载和优先生成。

### 2.4 Remini AI Photos

公开口径：

- 选择 8-12 张图片训练 AI。
- 选择 model image 来匹配姿势和风格。
- 生成不同姿势、角度和用途的人像。
- 强移动端，强调随时随地生成。

可借鉴：

- “样片图 / model image”可以成为墨境模板样片入口。
- 客户不需要理解 prompt，只需要选择想要的姿势和风格样张。
- 后续移动端或分享页很重要，但不是当前 最小可用版 首要任务。

### 2.5 WePics

公开口径：

- 上传两张脸照，选择风格，生成情侣和婚礼照片。
- 有 21 个集合、340+ 模板。
- 覆盖经典婚礼、旅拍、中式婚礼、情侣日常、婚登照、杂志风等。
- 提供带水印免费预览，高质量下载进入 Pro。
- 强调隐私和可删除。

可借鉴：

- 情侣/婚纱是高价值切口，不应只是单人写真旁支。
- 模板集合需要围绕“关系”和“场景”组织，而不是围绕模型技术组织。
- 免费预览加高清交付，是降低首次门槛的成熟模式。
- 婚登照、情侣日常、纪念日惊喜可以成为低成本传播场景。

### 2.6 POTO AI

公开口径：

- 主打无需实体拍摄，一张自拍即可生成中式婚礼照。
- 流程是选择风格、上传自拍、获得结果。
- 覆盖头像、肖像、婚礼照。
- 强调不训练照片，立即出结果。
- 数据保留和删除策略写在 FAQ。

可借鉴：

- “一张自拍即可试用”适合做低门槛体验入口。
- 但正式交付仍应保留后台资料审核和制作复核，避免成片不像。
- 中式婚礼、秀禾、汉服、旗袍可作为婚纱场景包重点。

### 2.7 Aragon / HeadshotPro

公开口径：

- 聚焦职业头像，先选择服装和背景，再上传少量照片。
- 生成 50+ 或最多 100 张头像。
- 结果可查看、编辑和下载。
- 团队头像场景强调统一风格、品牌一致性和批量交付。

可借鉴：

- 先选服装/背景，再上传，能降低用户焦虑。
- 对企业或多人场景，统一风格比单张惊艳更重要。
- 成片页必须支持收藏、返修、替换和下载。

## 3. 共同成功模式

成熟个人摄影棚产品基本都遵循这条链路：

```text
明确用途 -> 上传参考照 -> 创建身份资产 -> 选择场景包 -> 生成样片 -> 用户挑选 -> 批量成片 -> 下载/分享/复购
```

他们的核心优点：

| 优点 | 用户感受 | 墨境借鉴 |
|---|---|---|
| 摄影棚定位 | 我不是在学工具，是在拍照 | 前台改成“个人AI摄影棚” |
| 身份资产 | 训练一次，以后都能拍 | 参考照沉淀成“本人档案” |
| 场景包 | 不用写 prompt | 首页先选婚纱、情侣、闺蜜、儿童、女生写真、夕阳红 |
| 多人合照 | 情绪价值高 | 优先支持情侣、闺蜜、家庭和夕阳红双人 |
| 免费样片 | 首次尝试无压力 | 低清/带水印样片，确认后高清批量 |
| 质量承诺 | 知道好在哪里 | 像本人、皮肤真实、手自然、可打印、构图高级 |
| 隐私承诺 | 敢上传真人照 | 本地存储、可删除、不训练公共模型 |
| 交付闭环 | 像一次真实服务 | 复核、返修、替换、最终下载 |

## 4. 墨境主流程设计

### 4.1 客户平台主流程

```text
样片集 -> 注册/登录 -> 提交资料 -> 等待制作 -> 查看成片
```

客户平台是结果导向页面。客户只需要知道自己要交什么资料、当前是否需要补充、成片在哪里看。

### 4.2 样片集

样片集用于展示成片能力和可参考模板方向。

允许展示：

- 模板大类
- 子场景和样片组
- 适合人群
- 风格标签
- 开始定制入口

不展示：

- 上传表单
- 内部生产状态
- 过程稿
- 失败图
- 模板 ID
- 模型和接口参数

### 4.3 注册和登录

客户提交资料前必须注册或登录。

账号绑定：

- 定制订单
- 上传照片
- 制作状态
- 成片结果
- 下载和反馈记录

客户账号不具备后台权限，后台管理员账号单独管理。

### 4.4 提交资料

提交资料页只做 3 件事：

1. 选择影像组合。
2. 按人上传照片。
3. 勾选影像风格或模板方向。

常用组合：

- 单人女
- 单人男
- 情侣
- 闺蜜 2 人
- 闺蜜 3 人
- 全家福
- 其他

上传规则：

- 每个人最多 9 张照片。
- 多人订单按人分组上传。
- 客户可以写备注，但不填写内部参数。

主按钮统一为：

```text
提交定制需求
```

### 4.5 等待制作

客户提交后只看客户可理解状态：

- `已提交`
- `资料需补充`
- `制作中`
- `已完成`

客户不看到：

- `三视图`
- `人物确认图`
- `像不像审核`
- `后台上传`
- `脚本重跑`
- `复核标签`

这些动作都在后台制作平台处理。

### 4.6 查看成片

查看成片页只展示正式交付结果。

允许展示：

- 成片封面
- 成片列表
- 下载入口
- 成片数量
- 反馈入口
- 通知记录

不展示：

- 过程稿
- 失败图
- 被淘汰样片
- 工作人员备注
- 调试日志

### 4.7 后台制作平台主流程

```text
管理员登录 -> 接收订单 -> 审核资料 -> 制作成片 -> 上传成果 -> 通知客户 -> 完成交付
```

后台制作平台必须独立于客户平台。后台可以展示内部生产动作，但只能给管理员和工作人员使用。

后台核心模块：

- 订单列表
- 资料审核
- 制作记录
- 成果上传
- 通知与交付

正式成片仍以 `final_4k/` 为唯一交付目录。

## 5. 页面信息架构

### 5.0 当前前端导航口径

截至 2026-05-05，当前设计采用以下客户前台一级 / 二级业务板块：

1. `首页`
   - 无二级菜单
   - 讲生活愿景、服务价值和主入口
2. `影像定制`
   - `样片集`
   - `提交资料`
   - `制作状态`
   - `查看成片`
3. `图片创作`
   - `模板库`
   - `智能生图`
4. `我的作品`
   - `全部作品`
   - `已删除`
5. `登录`
   - 邮箱登录
   - 创建账号
6. `我的积分`
   - `我的积分`
   - `买积分`

后台制作平台是内部平台，不属于客户前台一级菜单：

```text
管理员登录 -> 接收订单 -> 审核资料 -> 制作成片 -> 上传成果 -> 通知客户 -> 完成交付
```

硬规则：

- 客户前端默认已连好，不展示 `连接设置`、`测试连接`。
- 6 个一级菜单是业务逻辑板块，不是界面排版规则；前端可响应式呈现。
- 后续如果业务设计要调整菜单数量、命名、顺序或归属，必须先改本文档并确认，再进入代码修改。
- `影像定制` 只承接客户定制订单，不展示内部生产动作。
- `图片创作` 只承接普通创作，不承接定制订单。
- `我的作品` 只承接普通生成历史、删除恢复和下载，不替代正式成片目录。
- `登录` 只承接登录、创建账号和账号归属，不承担连接配置职责。
- `我的积分` 独立承接积分、充值、订单和流水，不再混回定制流程。
- `后台制作平台` 只给管理员和工作人员使用，不能混入客户前台；后续单独开内网端口。

### 5.1 首页

影像定制首页默认是结果导向首页，只回答两个问题：

- 现在订单到哪一步了？
- 下一步要做什么？

推荐结构：

1. 顶部：产品名、当前项目、服务状态。
2. 样片区：大图样片、场景分类、风格标签。
3. 提交资料入口：选择组合、上传照片、勾选风格、提交。
4. 制作状态：已提交、资料需补充、制作中、已完成。
5. 结果区：只展示成片和下载入口。

### 5.2 场景包页

紧凑卡片网格：

- 每行 2-3 张卡。
- 每张卡 5 个字段以内。
- 主按钮只有一个。
- 不做大面积营销 hero。

### 5.3 资料中心

资料中心现在只作为后台或资料管理概念，不再作为客户前台主入口。

单人身份：

- 姓名或昵称
- 参考照
- 资料记录
- 状态
- 最近使用场景

组合身份：

- 情侣
- 闺蜜
- 家庭
- 夕阳红双人

组合资料必须能看到每个人的上传状态，避免多人订单时资料混乱。

### 5.4 图片创作里的智能生图

`图片创作 -> 智能生图` 的定位已经收敛为：

```text
直接创作入口，不是参数台。
```

在客户前台，`图片创作` 首页先回答“你是想快速玩风格，还是想做成套成片”。

因此首页表达先收口为两个主区块：

- `趣味生图`：单张、轻量、快速尝试
- `套图写真`：成组、统一风格、偏正式交付

`模板库` 继续存在，但退回次级入口、素材来源或筛选口径，不再抢客户第一认知。

固定主流程：

1. 上传照片
2. 补一句画面描述
3. 生成

辅助内容：

- 补充效果：换背景、换装、高清增强、局部修复
- 尺寸
- 质量
- 格式
- 数量
- 写法参考
- 模板库回跳
- 充值入口

硬规则：

- 上传照片和生成必须是首屏主体。
- 换背景、换装、高清增强、局部修复统一解释为 `补充效果`，是画面描述的补充，不改主流程定义。
- 右侧首位优先放 `补充效果`；提示词写法参考只作为轻量参考，放在右侧下方。
- 参数区默认后置，不能先于上传照片、画面描述和生成占据主视觉。
- 页面上只保留一个主动作，不把“模板库”“尺寸”“充值”做成并列主按钮。
- 如果是纯文字生成，也仍然按同一页处理，但页面层级不能因此倒回“先看参数、再找上传和生成”。

### 5.4 样片集页

样片集页用于展示模板大类、子场景、样片方向和风格参考。

结构：

- 上方：模板大类。
- 左侧：当前模板下的子场景和样片组。
- 右侧：大图、缩略图和放大预览。
- 模板大类平铺展示，避免横向滚动。
- 左侧样片组列表与右侧大图区域桌面端上下齐平。
- 样片图使用接近正方形的人像友好裁切，优先露出脸部。
- 点击左侧样片组或底部缩略图只切换同屏内容，不跳屏、不回弹、不打开额外详情屏。

不要把样片集和提交、生成、制作状态、成片交付混在一个列表里。

### 5.5 后台制作页

后台制作页是工作人员工作台，不是客户画廊。

紧凑表格字段：

- 订单号
- 客户
- 组合
- 状态
- 问题标签
- 操作

主动作：

- 审核资料
- 上传成果

同一页只保留一个主动作。

### 5.6 客户结果页

客户结果页只面向最终结果。

字段：

- 订单名
- 成片数量
- 最终图片
- 下载
- 完成状态
- 反馈入口

证据区折叠：

- 订单摘要
- 变更记录
- 通知记录

### 5.7 注册登录与我的积分

登录和我的积分已经从“设置”里拆开，作为客户前台一级菜单中的独立入口存在；后台制作平台不在客户前台展示。

登录：

- 只保留邮箱登录和创建账号
- 登录完成后直接回到主工作流
- 不再给客户暴露接口配置或测试动作

我的积分：

- 统一查看可用积分、积分明细、买积分和充值记录
- 买积分前必须先登录，未登录只提示先登录
- 后台制作平台单独给管理员和工作人员使用，不在客户菜单里混用
- 保持和创作主线同一套系统，不单独跳外部工具

内部接口参数、默认模型和调试能力可以保留在开发层，但不应再出现在客户前端菜单里。

### 5.8 客户平台与后台平台拆分

`影像定制` 只回答客户当前状态和下一步，不回答内部制作细节。

推荐拆法：

| 平台 | 只回答的问题 | 允许出现 | 不允许出现 | 主动作 |
| --- | --- | --- | --- | --- |
| `客户平台` | 现在该提交什么、是否需要补充、成片在哪里 | 样片集、资料提交、制作状态、结果展示 | 三视图、人物确认图、像不像审核、生图、后台上传 | `提交定制需求` |
| `后台制作平台` | 哪个订单要处理、资料是否通过、成果是否已上传 | 管理员登录、接收订单、审核资料、制作成片、上传成果、通知客户 | 客户营销文案、长故事、无关样片 | `上传成果` |

硬规则：

- 客户平台不展示内部生产动作。
- 客户平台必须绑定注册用户。
- 后台制作平台必须管理员登录，并通过独立内网端口访问。
- 成果只在后台上传，客户只看结果。

### 5.9 状态流拆分

客户看到的是单线结果状态，不是生产流程。

客户状态：

1. `待提交`
2. `已提交`
3. `资料需补充`
4. `制作中`
5. `已完成`

后台状态：

1. `新订单`
2. `待审核`
3. `需补传`
4. `待制作`
5. `制作中`
6. `待复核`
7. `待上传成果`
8. `待通知客户`
9. `已交付`

状态切换原则：

- 客户首页只显示 `当前状态 + 下一步`。
- 后台页可以显示更细状态，但不能原样透传到客户前台。
- 已完成步骤只保留摘要。
- 未开始步骤不抢当前主动作。

### 5.10 单一信息源

页面变复杂的另一个根因，是同一状态在多个区域重复出现。

后续前端实现必须强制单一信息源：

| 信息 | 单一来源 | 允许出现的位置 | 不允许重复出现的位置 |
| --- | --- | --- | --- |
| 当前订单 | 定制订单 | 顶部摘要、当前步骤卡 | Hero、场景大卡、交付卡同时重复 |
| 当前状态 | 订单状态 | 流程条、当前步骤卡 | 说明文案里重复再写一遍 |
| 资料状态 | 资料审核状态 | 提交资料页、顶部紧凑摘要 | 场景卡、交付卡重复写 |
| 当前风格方向 | 订单已选方向 | 提交资料页、顶部紧凑摘要 | 首页多处重复标签 |
| 可交付数量 | 交付模块统计 | 客户结果页、订单摘要 | 样片区、场景区重复写 |
| 下一步动作 | 当前状态机 | 首页主按钮、对应页面主按钮 | 多个区域同时出现多个主按钮 |

硬规则：

- 一个状态只有一个状态芯片。
- 一个阶段只有一个主按钮。
- 一个决定只在一个页面完成。

### 5.11 首页与二级页的切分原则

首页是调度器，二级页才是工作区。

首页负责：

- 告诉客户正在做什么
- 告诉客户当前进度
- 把客户送到下一步

二级页负责：

- 展示该步骤的完整细节
- 承担该步骤的唯一决策
- 保存该步骤的证据

因此首页不应出现：

- 全量场景卡网格
- 完整资料审核表单
- 完整样片方向列表
- 成片交付大列表

首页只应保留：

- 当前场景摘要
- 当前步骤摘要
- 下一步按钮
- 紧凑项目摘要

### 5.12 影像定制首屏口径

首屏一句话只允许表达：

```text
你现在在做什么，以及下一步点哪里。
```

不再建议首屏同时出现：

- `生图流程`
- `客户工作流`
- `生图流程一共四步`
- `客户按 4 个区完成选择`
- `当前任务`

这些标题语义过近，放在同一屏会互相抢层级。

推荐压成：

- 主标题：`影像定制`
- 副标题：`提交照片和影像方向，我们制作完成后通知你看成片`
- 当前步骤标题：`现在先完成这一步`

## 7. 最小可用版 分期

### 7.1 最小可用版 0：设计和数据口径

目标：

- 统一“一个订单，两套平台”口径。
- 明确客户账号、后台账号、定制订单、成果交付的数据边界。

交付：

- 本文档。
- 定制流程 SOP。
- 客户状态字段表。
- 后台状态字段表。

### 7.2 最小可用版 1：客户平台壳

目标：

- 让注册用户能提交资料并查看结果。

功能：

- 注册和登录。
- 样片集。
- 提交资料。
- 制作状态。
- 查看成片。

验收：

- 客户不需要看到内部生产词，也能完成定制提交。
- 未登录不能提交定制订单和查看个人成片。

### 7.3 最小可用版 2：后台制作平台壳

目标：

- 让管理员能接单、审核资料并上传成果。

功能：

- 管理员登录。
- 订单列表。
- 资料审核。
- 成果上传。
- 通知客户。

验收：

- 后台入口和客户入口分开。
- 管理员能围绕同一个订单上传正式成果。

### 7.4 最小可用版 3：交付闭环

目标：

- 从“上传结果”升级为“可追踪交付”。

功能：

- 成果版本记录。
- 下载记录。
- 邮件或站内通知。
- 客户反馈。

验收：

- 客户结果页不出现失败图和过程稿。
- 成果数量和后台上传数量一致。

### 7.5 最小可用版 4：多人组合

目标：

- 支持高价值关系场景。

功能：

- 情侣订单。
- 闺蜜订单。
- 家庭订单。
- 每人最多 9 张照片。

验收：

- 每个人的照片按人归档。
- 多人订单能清楚展示人物数量和资料状态。

### 7.6 最小可用版 5：后台质量门禁

目标：

- 降低返工成本。

功能：

- 资料审核。
- 制作复核。
- 返修记录。
- 成果版本替换。

验收：

- 客户前台只看到 `资料需补充`、`制作中`、`已完成` 等结果状态。
- 后台能追踪问题原因，但不把内部细节透传给客户。

## 9. 文案口径

### 9.1 推荐产品名

```text
墨境个人AI摄影棚
```

### 9.2 一句话说明

```text
客户提交照片和影像方向，后台制作并上传成果，客户只查看可交付成片。
```

### 9.3 核心卖点

- 好选择：按场景包提交，不用写提示词。
- 可交付：订单、审核、制作、上传、下载分开管理。
- 支持多人：情侣、闺蜜、家庭和夕阳红双人照。
- 更安心：本地记录、可删除、证据可追溯。

### 9.4 禁用表达

避免在客户前台使用：

- 模型试错
- prompt 调试
- API 中转
- gpt-image-2 参数
- 失败重跑
- 模板 ID

这些内容可以保留在高级设置或证据区。

## 10. 风险与应对

| 风险 | 表现 | 应对 |
|---|---|---|
| 资料不全 | 客户提交后无法制作 | 提交前做资料提示，后台审核补传 |
| 模板太多 | 用户不知道选什么 | 首页只放场景包，完整库折叠 |
| 多人资料混乱 | 情侣/闺蜜身份混乱 | 每个人独立照片归档，组合状态可见 |
| 批量成本高 | 大量返修 | 后台审核后再制作 |
| 客户担心隐私 | 不敢上传真人照 | 本地存储、可删除、用途说明前置 |
| 页面太散 | 像管理后台而不是产品 | 客户页一个主动作，后台页独立处理 |
| 交付混乱 | 过程稿、失败图、成片混在一起 | 客户结果页、后台制作页、后台交付页分离 |

## 11. UI 紧凑度规则

- 首屏不做大 hero，不讲长故事。
- 顶部先给项目、状态、下一步。
- 场景卡不超过 5 个字段。
- 样片卡不展示完整 prompt。
- 表格优先展示预览、状态和操作，不堆说明文字。
- 历史、失败、日志和完整模板全部折叠。
- 同一状态只出现一次，不在多个区域重复。
- 每个页面只保留一个主动作。
