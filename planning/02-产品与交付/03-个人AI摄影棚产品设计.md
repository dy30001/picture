## 1. 产品定位

墨境下一阶段的定位应从：

```text
GPT Image 2 本地图像生成工作台
```

升级为：

```text
个人AI摄影棚：上传本人参考照，确认数字分身，再按场景包生成写真、婚纱、情侣、闺蜜、儿童、夕阳红等成片。
```

核心判断：

- 竞品强在“用户感觉自己拥有了一个随叫随到的摄影棚”。
- 墨境不能只展示 2300+ prompt 模板，否则用户会觉得自己还在操作工具。
- 墨境前台应拆成两个清楚功能：`样片集` 和 `生图`。
- `样片集` 是独立浏览入口，先让用户看到好看的样片、风格和可拍方向。
- `生图` 是正式操作流程，固定为：传三张图 -> 确认像本人 -> 选模板参考 -> 正式生成。
- 墨境的差异化不在“比别人模板更多”，而在“半专业交付流程”：身份确认门禁、模板参考、批量复核、返修记录、最终交付。

## 1.1 当前产品结构更新

用户最新确认的产品结构：

```text
一级功能 A：样片集
- 用于浏览效果、判断审美、选择想参考的拍摄方向。
- 样片集不是生图流程的一步，不承担上传和生成动作。
- 样片集必须好看、直观、像摄影棚作品集。
- 样片集内部只做一屏交互：先选大场景，再看样片组矩阵，当前组内照片直接在同屏预览。
- 具体样片优先按 `final_4k/` 子目录自动成组，不能只展示少量固定示例图，也不能把不同目录的主题照片混在一个方向里。

一级功能 B：生图
- 第 1 步：传三张图。
- 第 2 步：确认像本人。
- 第 3 步：选模板，可参考某个模板来生成。
- 第 4 步：正式生成。
```

界面口径：

- `样片集` 只展示样片、场景、样片组和组内照片。
- `生图` 只展示当前生成进度、上传入口、身份确认、模板选择和生成动作。
- 模板是生成参考，不再和样片集、身份确认混在同一个区域里。

## 2. 竞品模式学习

### 2.1 Photo AI

公开口径：

- 定位为 AI Photographer。
- 用户上传自拍，创建自己的 AI model。
- 创建 AI model 后，可以在任何姿势、地点或动作中拍照。
- 支持 100s of photo packs。
- 支持多人、视频、换装、产品视频。
- 页面强调训练一次后可快速生成无限照片。

可借鉴：

- 用“摄影师 / 摄影棚”表达产品，而不是“生成器”。
- 把身份建模当作资产，而不是一次性上传。
- 用 photo packs 降低 prompt 门槛。
- 多人、视频、换装可以作为后续增长功能。

不直接照搬：

- 他们更偏自助 SaaS；墨境更适合做成“服务交付工作台 + 客户选片台”。
- 当前阶段不做完整模型训练，优先沿用参考照输入和身份确认门禁。

### 2.2 pic.mx

公开口径：

- 定位为 AI Photo Studio 和 personal AI photographer。
- 用户上传自拍，获得个人 AI model。
- 强调任意风格、任意地点、秒级出图。
- 支持情侣、朋友、家庭多人模型合照。
- 强调隐私：模型只属于用户，可删除。
- 对比传统摄影棚：成本低、速度快、不需要预约。

可借鉴：

- “个人AI摄影棚”是最适合墨境的前台表达。
- 多人模型组合非常适合墨境的情侣照、闺蜜照、夕阳红和家庭照。
- 隐私承诺必须前置，尤其是真人参考照上传场景。
- 和传统摄影棚做对比时，重点说省预约、省拍摄、省选景、省返工。

### 2.3 PhotosAI

公开口径：

- 用户上传 10-20 张自拍。
- 训练个人 AI model 约 5 分钟。
- 按 LinkedIn、Instagram、Tinder、Travel、Fashion 等 photo packs 使用。
- 强调高相似度、真实皮肤、高分辨率、手部细节、数据安全。
- 订阅套餐按照片数、模型数、清晰度、优先生成等分级。

可借鉴：

- 上传环节必须给明确数量和质量标准。
- 场景包要按真实用途命名，不按内部风格命名。
- 质量卖点不能只说“高清”，还要说“像本人、皮肤真实、手部自然、可打印”。
- 套餐分级可以围绕身份数、场景数、成片数、高清下载和优先生成。

### 2.4 Remini AI Photos

公开口径：

- 选择 8-12 张图片训练 AI。
- 选择 model image 来匹配姿势和风格。
- 生成不同姿势、角度和用途的人像。
- 强移动端，强调随时随地生成。

可借鉴：

- “样片图 / model image”可以成为墨境模板样片入口。
- 客户不需要理解 prompt，只需要选择想要的姿势和风格样张。
- 后续移动端或分享页很重要，但不是当前 最小可用版 首要任务。

### 2.5 WePics

公开口径：

- 上传两张脸照，选择风格，生成情侣和婚礼照片。
- 有 21 个集合、340+ 模板。
- 覆盖经典婚礼、旅拍、中式婚礼、情侣日常、婚登照、杂志风等。
- 提供带水印免费预览，高质量下载进入 Pro。
- 强调隐私和可删除。

可借鉴：

- 情侣/婚纱是高价值切口，不应只是单人写真旁支。
- 模板集合需要围绕“关系”和“场景”组织，而不是围绕模型技术组织。
- 免费预览加高清交付，是降低首次门槛的成熟模式。
- 婚登照、情侣日常、纪念日惊喜可以成为低成本传播场景。

### 2.6 POTO AI

公开口径：

- 主打无需实体拍摄，一张自拍即可生成中式婚礼照。
- 流程是选择风格、上传自拍、获得结果。
- 覆盖头像、肖像、婚礼照。
- 强调不训练照片，立即出结果。
- 数据保留和删除策略写在 FAQ。

可借鉴：

- “一张自拍即可试用”适合做低门槛体验入口。
- 但正式交付仍应保留墨境身份确认，避免成片不像。
- 中式婚礼、秀禾、汉服、旗袍可作为婚纱场景包重点。

### 2.7 Aragon / HeadshotPro

公开口径：

- 聚焦职业头像，先选择服装和背景，再上传少量照片。
- 生成 50+ 或最多 100 张头像。
- 结果可查看、编辑和下载。
- 团队头像场景强调统一风格、品牌一致性和批量交付。

可借鉴：

- 先选服装/背景，再上传，能降低用户焦虑。
- 对企业或多人场景，统一风格比单张惊艳更重要。
- 成片页必须支持收藏、返修、替换和下载。

## 3. 共同成功模式

成熟个人摄影棚产品基本都遵循这条链路：

```text
明确用途 -> 上传参考照 -> 创建身份资产 -> 选择场景包 -> 生成样片 -> 用户挑选 -> 批量成片 -> 下载/分享/复购
```

他们的核心优点：

| 优点 | 用户感受 | 墨境借鉴 |
|---|---|---|
| 摄影棚定位 | 我不是在学工具，是在拍照 | 前台改成“个人AI摄影棚” |
| 身份资产 | 训练一次，以后都能拍 | 参考照沉淀成“本人档案” |
| 场景包 | 不用写 prompt | 首页先选婚纱、情侣、闺蜜、儿童、女生写真、夕阳红 |
| 多人合照 | 情绪价值高 | 优先支持情侣、闺蜜、家庭和夕阳红双人 |
| 免费样片 | 首次尝试无压力 | 低清/带水印样片，确认后高清批量 |
| 质量承诺 | 知道好在哪里 | 像本人、皮肤真实、手自然、可打印、构图高级 |
| 隐私承诺 | 敢上传真人照 | 本地存储、可删除、不训练公共模型 |
| 交付闭环 | 像一次真实服务 | 复核、返修、替换、最终下载 |

## 4. 墨境主流程设计

### 4.1 客户主流程

```text
A. 样片集
1. 浏览样片
2. 切换场景
3. 选择可参考方向

B. 生图
1. 传三张图
2. 确认像本人
3. 选模板作为生成参考
4. 正式生成
```

样片集和生图是两个一级功能。样片集负责展示效果和审美方向，不是生图门禁；生图只按四步推进，未传满三张图、未完成身份确认或未选模板时，不开放正式生成。

### 4.2 步骤一：选择场景

首批场景：

- 婚纱照
- 情侣照
- 闺蜜照
- 儿童 10 岁照
- 女生写真
- 夕阳红

每个场景卡只展示：

- 场景名称
- 适合人群
- 推荐张数
- 是否支持多人
- 当前可用状态
- 主按钮

不要展示：

- 模板 ID
- 长 prompt
- 模型名
- 接口模式
- 内部参数

### 4.3 步骤二：上传参考照

参考照要求：

- 正脸、半身、全身至少各 1 张。
- 不同光线和角度，避免全部同一场景。
- 不要严重磨皮、遮挡、过暗、过曝。
- 情侣、闺蜜、家庭场景要分别上传每个人的参考照。

最小可用版 可先做人工检查，后续再做自动体检。

参考照体检项：

- 脸是否清晰
- 角度是否足够
- 光线是否可用
- 是否戴墨镜或口罩
- 是否过度美颜
- 多人照是否能分清每个人

### 4.4 步骤三：确认像本人

把内部身份门禁改成客户能理解的“确认像本人”。

状态：

- `待上传`
- `待生成`
- `待确认`
- `已确认`
- `需返修`

通过标准：

- 脸像本人。
- 年龄感不漂移。
- 发型和气质不明显跑偏。
- 身材比例不异常。
- 情侣/多人之间身份不串脸。

规则：

- 未完成身份确认时，不开放批量成片主动作。
- 身份确认通过后，才允许进入模板选择和正式生成。
- 参考照证据用缩略图展示，详细记录折叠。

### 4.5 独立功能：样片集

样片不是生图步骤，也不是最终交付，而是独立样片集，用来展示效果、风格和可参考方向。

每个场景默认推荐 3-6 个方向：

- 婚纱照：中式、旅拍、韩式、杂志、婚登、复古。
- 情侣照：日常、旅行、纪念日、校园、都市、电影感。
- 闺蜜照：棚拍、街拍、生日、旅行、学院、复古。
- 儿童 10 岁照：校园、生日、成长纪念、户外、亲子。
- 女生写真：通勤、法式、胶片、国风、杂志、旅行。
- 夕阳红：纪念照、旅行、合照、旗袍、家庭、怀旧。

样片卡字段：

- 标题
- 适合人群
- 关键词
- 推荐张数
- 使用按钮

完整模板库放到“更多模板”，默认折叠。

### 4.6 步骤五：批量成片

批量成片只在两个条件满足后启动：

- 身份已确认。
- 样片方向已确认。

批量任务字段：

- 编号
- 场景
- 身份组合
- 样片方向
- 状态
- 预览
- 复核标签
- 操作

状态：

- `排队中`
- `生成中`
- `待复核`
- `需返修`
- `可交付`

### 4.7 步骤六：复核与交付

复核标签：

- 脸不像
- 年龄不对
- 身材比例异常
- 手部异常
- 服装不对
- 背景不高级
- 裁切不完整
- 清晰度不足
- 多人串脸

交付页只展示：

- 最终图片
- 下载入口
- 交付数量核对
- 返修记录摘要
- 验收状态

不展示：

- 失败图
- 被替换图
- 样片图
- 调试日志

## 5. 页面信息架构

### 5.0 当前前端导航口径

截至 2026-05-03，当前设计采用以下一级 / 二级业务板块：

1. `拍摄定制`
   - `样片集`
   - `生图`
   - `身份确认`
   - `选模板`
   - `生成`
2. `图片创作`
   - `模板库`
   - `智能生图`
   - 客户前台主区推荐收口为 `趣味生图` / `套图写真` 两个区块；`模板库`、`智能生图` 继续保留为现阶段路由和能力口径
3. `我的作品`
   - `全部作品`
   - `已删除`
4. `注册开通`
   - `注册`
   - `登录`
5. `我的积分`
   - `我的积分`
   - `充值中心`
   - `充值订单`
   - `积分流水`

硬规则：

- 客户前端默认已连好，不展示 `连接设置`、`测试连接`。
- 5 个一级菜单是业务逻辑板块，不是界面排版规则；前端可响应式呈现。
- 后续如果业务设计要调整菜单数量、命名、顺序或归属，必须先改本文档并确认，再进入代码修改。
- `图片创作` 只保留客户真正会走的动作，不把接口配置做成二级菜单；新增编辑类入口必须落到可操作界面，不能只做空菜单。
- `我的作品` 只承接普通生成历史、删除恢复和下载，不替代正式成片目录。
- `注册开通` 只承接注册、登录和账号归属，不承担连接配置职责。
- `我的积分` 独立承接积分、充值、订单和流水，不再混回图片创作流程。

### 5.1 首页

拍摄定制首页默认是 `样片集`，只回答两个问题：

- 有哪些好看的样片方向？
- 要不要开始生图？

推荐结构：

1. 顶部：产品名、当前项目、服务状态。
2. 样片集：大图样片、场景分类、风格标签。
3. 生图入口：单一主按钮进入四步流程。
4. 生图页：只展示传三张图、身份确认、选模板、生成。
5. 证据区：默认折叠。

### 5.2 场景包页

紧凑卡片网格：

- 每行 2-3 张卡。
- 每张卡 5 个字段以内。
- 主按钮只有一个。
- 不做大面积营销 hero。

### 5.3 身份中心

身份中心管理“人”和“组合”。

单人身份：

- 姓名或昵称
- 参考照
- 身份确认
- 状态
- 最近使用场景

组合身份：

- 情侣
- 闺蜜
- 家庭
- 夕阳红双人

组合身份必须能看到每个人的底片状态，避免多人批量时身份混乱。

### 5.4 图片创作里的智能生图

`图片创作 -> 智能生图` 的定位已经收敛为：

```text
直接创作入口，不是参数台。
```

在客户前台，`图片创作` 首页先回答“你是想快速玩风格，还是想做成套成片”。

因此首页表达先收口为两个主区块：

- `趣味生图`：单张、轻量、快速尝试
- `套图写真`：成组、统一风格、偏正式交付

`模板库` 继续存在，但退回次级入口、素材来源或筛选口径，不再抢客户第一认知。

固定主流程：

1. 上传照片
2. 补一句画面描述
3. 开拍

辅助内容：

- 可选功能
- 尺寸
- 质量
- 格式
- 数量
- 模板库回跳
- 充值入口

硬规则：

- 上传照片和开拍必须是首屏主体。
- 可选功能只解释为“补充效果”或“追加描述”，不改主流程定义。
- 提示词写法参考是辅助说明，放侧边辅助区；主区只保留“上传照片、写提示词、开拍”。
- 参数区默认后置，不能先于上传照片和开拍占据主视觉。
- 页面上只保留一个主动作，不把“模板库”“尺寸”“充值”做成并列主按钮。
- 如果是纯文字生成，也仍然按同一页处理，但页面层级不能因此倒回“先看参数、再找上传和开拍”。

### 5.4 样片页

样片页用于比较方向。

结构：

- 左侧：已确认身份和当前场景。
- 中间：推荐样片方向。
- 右侧或底部：已选方向和下一步。

不要把样片和最终成片混在一个列表里。

### 5.5 批量成片页

批量页是工作台，不是画廊。

紧凑表格字段：

- 预览
- 编号
- 场景
- 身份
- 状态
- 问题标签
- 操作

主动作：

- 生成下一批
- 或提交交付

同一页只保留一个主动作。

### 5.6 交付页

交付页只面向最终结果。

字段：

- 项目名
- 交付数量
- 最终图片
- 下载
- 验收状态
- 返修摘要

证据区折叠：

- 原始参考照
- 身份确认
- 样片记录
- 返修记录
- 生成日志

### 5.7 注册登录与我的积分

注册开通和我的积分已经从“设置”里拆开，作为 5 个一级菜单中的独立入口存在。

注册开通：

- 只保留 `注册` 和 `登录`
- 注册完成后直接回到主工作流
- 不再给客户暴露接口配置或测试动作

我的积分：

- 统一查看可用积分、充值中心、充值订单、积分流水
- 保持和创作主线同一套系统，不单独跳外部工具

内部接口参数、默认模型和调试能力可以保留在开发层，但不应再出现在客户前端菜单里。

### 5.8 拍摄定制逻辑拆分

`拍摄定制` 不是一个“大而全首页”，而是一个总调度入口。

必须拆成 4 类不同问题：

1. `选什么服务`
   - 由 `场景套餐` 回答
2. `像不像本人`
   - 由 `身份确认` 回答
3. `走哪个成片方向`
   - 由 `样片确认` 回答
4. `哪些图最终交付`
   - 由 `成片交付` 回答

如果一个区域同时试图回答以上 2 个以上问题，页面就会开始变复杂。

推荐拆法：

| 区域 | 只回答的问题 | 允许出现 | 不允许出现 | 主动作 |
| --- | --- | --- | --- | --- |
| `样片集` | 有哪些好看的参考方向 | 大图样片、场景分类、风格标签 | 上传、身份确认、生成表单 | `开始生图` |
| `生图` | 现在到哪一步、下一步是什么 | 四步流程、当前步骤、下一步按钮、紧凑摘要 | 完整样片列表、模板库全集、交付明细 | `下一步` |
| `身份确认` | 三张图是否到位，身份是否确认 | 上传入口、已传数量、身份状态、返修动作 | 样片全集、模板列表、正式生成按钮 | `上传三张图` / `确认像本人` |
| `选模板` | 用哪个模板作为生成参考 | 已选模板摘要、进入模板库按钮 | 样片全集、上传入口、交付列表 | `去模板库选模板` |
| `生成` | 是否满足正式生成条件 | 已传图数、模板状态、前置完成数、生成按钮 | 样片确认、身份返修细节 | `进入正式生成` |

硬规则：

- `样片集` 不能承担上传和生成。
- `生图` 不能把样片浏览当作第 1 步。
- `身份确认` 必须要求三张图。
- `选模板` 必须在身份确认后开放。
- `生成` 必须在模板选择后开放。

### 5.9 状态流拆分

客户看到的流程，应该是单线状态流，不是“所有步骤同时可见”。

推荐状态：

1. `未选场景`
2. `待上传参考照`
3. `待确认身份`
4. `待选择模板`
5. `待正式生成`
6. `成片生成中`
7. `待交付复核`
8. `已交付`

每个状态都要绑定唯一页面重心：

| 状态 | 首页显示重点 | 应该展开的页 | 默认折叠 |
| --- | --- | --- | --- |
| `未选场景` | 当前还未开始，先选场景 | `场景套餐` | 身份确认、样片、交付 |
| `待上传参考照` | 当前场景已定，先传参考照 | `身份确认` | 样片、交付 |
| `待确认身份` | 先确认像本人 | `身份确认` | 样片集、交付 |
| `待选择模板` | 身份已确认，选择生成参考 | `选模板` | 样片集、交付 |
| `待正式生成` | 模板已定，可以正式生成 | `生成` | 身份确认明细、全量样片 |
| `成片生成中` | 展示进度和预计结果 | `生成` | 前序输入表单 |
| `待交付复核` | 只看最终候选和问题标签 | `生成` | 样片与参考照 |
| `已交付` | 只看最终结果和下载 | `生成` | 失败图、样片、调试记录 |

状态切换原则：

- 首页只显示“当前状态 + 下一步”。
- 已完成步骤只保留摘要，不再展开完整内容。
- 未开始步骤只显示为后续，不抢当前主动作。

### 5.10 单一信息源

页面变复杂的另一个根因，是同一状态在多个区域重复出现。

后续前端实现必须强制单一信息源：

| 信息 | 单一来源 | 允许出现的位置 | 不允许重复出现的位置 |
| --- | --- | --- | --- |
| 当前场景 | 项目当前场景 | 顶部摘要、当前步骤卡 | Hero、场景大卡、交付卡同时重复 |
| 当前步骤 | 工作流状态 | 流程条、当前步骤卡 | 说明文案里重复再写一遍 |
| 身份状态 | 身份确认状态 | `身份确认` 卡、顶部紧凑摘要 | 场景卡、交付卡重复写 |
| 当前样片方向 | 样片确认状态 | `样片确认` 页、顶部紧凑摘要 | 首页多处重复标签 |
| 可交付数量 | 交付模块统计 | `成片交付` 页、项目摘要 | 样片区、场景区重复写 |
| 下一步动作 | 当前步骤机 | 首页主按钮、对应页面主按钮 | 多个区域同时出现多个主按钮 |

硬规则：

- 一个状态只有一个状态芯片。
- 一个阶段只有一个主按钮。
- 一个决定只在一个页面完成。

### 5.11 首页与二级页的切分原则

首页是调度器，二级页才是工作区。

首页负责：

- 告诉客户正在做什么
- 告诉客户当前进度
- 把客户送到下一步

二级页负责：

- 展示该步骤的完整细节
- 承担该步骤的唯一决策
- 保存该步骤的证据

因此首页不应出现：

- 全量场景卡网格
- 完整身份确认表单
- 完整样片方向列表
- 成片交付大列表

首页只应保留：

- 当前场景摘要
- 当前步骤摘要
- 下一步按钮
- 紧凑项目摘要

### 5.12 拍摄定制首屏口径

首屏一句话只允许表达：

```text
你现在在做什么，以及下一步点哪里。
```

不再建议首屏同时出现：

- `生图流程`
- `客户工作流`
- `生图流程一共四步`
- `客户按 4 个区完成选择`
- `当前任务`

这些标题语义过近，放在同一屏会互相抢层级。

推荐压成：

- 主标题：`拍摄定制`
- 副标题：`先选场景，再按步骤确认，最后交付成片`
- 当前步骤标题：`现在先完成这一步`

## 7. 最小可用版 分期

### 7.1 最小可用版 0：设计和数据口径

目标：

- 统一“个人AI摄影棚”口径。
- 明确场景、身份、样片、成片、交付的数据边界。

交付：

- 本文档。
- 场景包字段表。
- 身份状态字段表。
- 复核标签表。

### 7.2 最小可用版 1：客户流程壳

目标：

- 不改底层生成能力，先把现有工具包装成摄影棚流程。

功能：

- 首页流程条。
- 场景包入口。
- 身份确认页。
- 样片页。
- 批量成片页。
- 交付页。

验收：

- 用户不需要看到模型名和 prompt，也能走完整流程。
- 未完成身份确认时，不能启动批量成片。

### 7.3 最小可用版 2：场景包产品化

目标：

- 把已有模板变成可卖的场景包。

首批场景：

- 婚纱照
- 情侣照
- 闺蜜照
- 儿童 10 岁照
- 女生写真
- 夕阳红

验收：

- 每个场景至少 3 个样片方向。
- 每个方向有推荐张数、关键词和客户可读说明。
- 完整模板库默认折叠。

### 7.4 最小可用版 3：复核和交付

目标：

- 从“生成图片”升级为“交付服务”。

功能：

- 成片复核标签。
- 可交付筛选。
- 替换/返修记录。
- 最终下载包。

验收：

- 交付页不出现失败图和样片图。
- 每张被剔除图片都有结构化原因。
- 最终数量和套餐数量一致。

### 7.5 最小可用版 4：多人组合

目标：

- 支持高价值关系场景。

功能：

- 情侣身份组合。
- 闺蜜身份组合。
- 家庭身份组合。
- 夕阳红双人组合。

验收：

- 每个人都有独立身份确认状态。
- 批量成片能识别身份组合。
- 复核标签支持“多人串脸”。

### 7.6 最小可用版 5：自动质量门禁

目标：

- 降低人工返修成本。

功能：

- 参考照体检。
- 出图质量初筛。
- 相似度辅助评分。
- 手部和裁切异常提示。

验收：

- 上传阶段能提示明显不合格参考照。
- 复核阶段能自动标记可疑图，但最终仍保留人工确认。

## 9. 文案口径

### 9.1 推荐产品名

```text
墨境个人AI摄影棚
```

### 9.2 一句话说明

```text
上传本人参考照，确认像本人，再按婚纱、情侣、闺蜜、儿童、写真、夕阳红等场景生成可交付成片。
```

### 9.3 核心卖点

- 像本人：先确认像本人，再生成样片。
- 好选择：按场景包拍照，不用写提示词。
- 可交付：样片、复核、返修、下载分开管理。
- 支持多人：情侣、闺蜜、家庭和夕阳红双人照。
- 更安心：本地记录、可删除、证据可追溯。

### 9.4 禁用表达

避免在客户前台使用：

- 模型试错
- prompt 调试
- API 中转
- gpt-image-2 参数
- 失败重跑
- 模板 ID

这些内容可以保留在高级设置或证据区。

## 10. 风险与应对

| 风险 | 表现 | 应对 |
|---|---|---|
| 身份不像 | 客户拒绝全部成片 | 身份确认必须前置 |
| 模板太多 | 用户不知道选什么 | 首页只放场景包，完整库折叠 |
| 多人串脸 | 情侣/闺蜜身份混乱 | 每个人独立底片，组合状态可见 |
| 批量成本高 | 大量返修 | 样片确认后再批量 |
| 客户担心隐私 | 不敢上传真人照 | 本地存储、可删除、用途说明前置 |
| 页面太散 | 像管理后台而不是产品 | 单页一个主动作，状态单一来源 |
| 交付混乱 | 样片、失败图、成片混在一起 | 样片页、批量页、交付页分离 |

## 11. UI 紧凑度规则

- 首屏不做大 hero，不讲长故事。
- 顶部先给项目、状态、下一步。
- 场景卡不超过 5 个字段。
- 样片卡不展示完整 prompt。
- 表格优先展示预览、状态和操作，不堆说明文字。
- 历史、失败、日志和完整模板全部折叠。
- 同一状态只出现一次，不在多个区域重复。
- 每个页面只保留一个主动作。
