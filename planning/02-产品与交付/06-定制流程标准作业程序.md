# 定制流程标准作业程序

日期： 2026-05-04
模式： doc
工作区： `/Users/dy3000/code/pic`

## 0. 当前结论

影像定制从现在起按“一个订单，两套平台”设计：

```text
客户平台：样片集 -> 提交资料 -> 等待制作 -> 查看成片
后台制作平台：管理员登录 -> 接收订单 -> 审核资料 -> 制作成片 -> 上传成果 -> 通知客户 -> 完成交付
```

核心变化：

- 客户只关注结果，不参与生产过程。
- 客户平台必须考虑注册用户；提交资料、查看成片、下载和反馈都绑定登录用户。
- 后台制作平台只给管理员和工作人员使用。
- `三视图`、`人物确认图`、`像不像审核`、`后台上传` 都是内部生产动作，不出现在客户主流程。
- 样片只负责让客户理解最终效果，不是定制订单的强制步骤。
- 正式交付仍以 `final_4k/` 为唯一成片目录。

## 1. 适用范围

这份标准流程适用于以下定制型项目：

- 婚纱旅拍
- 情侣写真
- 闺蜜写真
- 儿童写真
- 地标人像
- 旅行人像
- 全家福和其他多人定制

它不解决“开放式自由生图”场景，只解决“真人照片驱动、后台制作、客户查看结果”的定制项目。

## 2. 一句话主流程

客户视角只需要理解 4 个动作：

```text
看样片 -> 提交资料 -> 等待制作 -> 查看成片
```

后台视角必须完成 7 个动作：

```text
管理员登录 -> 接收订单 -> 审核资料 -> 制作成片 -> 上传成果 -> 通知客户 -> 完成交付
```

主对象是 `定制订单`。客户提交订单，后台围绕订单制作和上传成果，客户最终只看到状态和成片。

## 3. 客户平台流程

### 3.1 看样片

样片用于展示效果、审美和可拍方向。

客户看到：

- 场景分类
- 样片组
- 适合人群
- 推荐风格
- 开始定制入口

客户不看到：

- 生成过程
- 中间图
- 内部模板 ID
- 失败图
- 调试记录

### 3.2 注册或登录

客户提交资料前必须有账号。

需要账号的动作：

- 创建定制订单
- 上传本人照片
- 查看订单状态
- 查看和下载成片
- 提交反馈

客户账号只承担客户侧身份，不等同于管理员账号。

### 3.3 提交资料

客户提交资料时只做 3 件事：

1. 选择影像组合。
2. 按人上传照片。
3. 勾选想要的影像风格或模板方向。

常用影像组合：

- 单人女
- 单人男
- 情侣
- 闺蜜 2 人
- 闺蜜 3 人
- 全家福
- 其他

上传规则：

- 按“人”上传，不按订单混传。
- 每个人最多上传 9 张照片。
- 多人订单必须能分清每个人。
- 客户可以补充备注，但不需要填写内部参数。

提交按钮统一写：

```text
提交定制需求
```

### 3.4 等待制作

客户提交后，前台只展示结果导向状态。

客户可见状态：

- `待提交`
- `已提交`
- `资料需补充`
- `制作中`
- `已完成`

客户不需要知道：

- 工作人员是否生成了中间图
- 是否做了像不像审核
- 哪个模型或脚本在跑
- 哪张过程图被淘汰

### 3.5 查看成片

成片完成后，客户通过站内结果页查看。

客户可做：

- 查看成片
- 下载成片
- 提交反馈
- 查看通知记录

客户不看到：

- `generated/`
- `review/`
- 失败图
- 旧版本过程稿
- 工作人员备注

## 4. 后台制作平台流程

### 4.1 管理员登录

后台必须独立登录，只允许管理员或工作人员进入。

后台账号至少要区分：

- 管理员
- 制作人员
- 复核人员

第一阶段可以先做单一管理员角色，但入口和权限口径必须按后台平台设计。

### 4.2 接收订单

后台订单列表至少展示：

- 订单编号
- 客户账号
- 影像组合
- 人物数量
- 已上传照片数
- 已选风格
- 客户备注
- 当前状态
- 创建时间

后台订单状态不直接等同于客户可见状态，可以更细。

### 4.3 审核资料

工作人员审核客户上传资料。

审核项：

- 每个人照片是否足够
- 脸是否清楚
- 遮挡是否太多
- 光线是否可用
- 多人订单是否能分清身份
- 风格选择是否明确

审核结果：

- `资料通过`
- `需要补传`
- `需要客服沟通`

需要补传时，后台发通知，客户前台只看到 `资料需补充`。

### 4.4 制作成片

制作人员在后台处理所有生产动作。

内部可包含：

- 照片整理
- 身份稳定性检查
- 参考图筛选
- 中间图生成
- 批量成片
- 返修
- 复核

这些都是后台动作，不作为客户前台步骤。

批量成片仍可直接生成 4K 候选图。只有复核通过并进入 `final_4k/` 后，才算正式成片。

### 4.5 上传成果

后台必须提供成果上传入口。

上传对象：

- 正式成片
- 版本说明
- 成片数量
- 可选封面图

上传规则：

- 只把可交付图片放进客户结果页。
- 过程稿、失败图、淘汰图不进入客户结果页。
- 正式成片目录继续以 `final_4k/` 为准。

### 4.6 通知客户

后台可以触发邮件或站内通知。

通知场景：

- 资料需要补充
- 订单开始制作
- 成片已完成
- 成片已更新

邮件文案只说客户需要做什么，不讲内部生产细节。

### 4.7 完成交付

订单交付完成前必须核对：

- 客户结果页可打开
- 成片数量正确
- 下载可用
- 通知已发出
- 订单状态已更新

## 5. 三个硬门禁

### 5.1 账号门禁

客户未注册或未登录前：

- 不能提交定制订单
- 不能上传私人照片
- 不能查看个人成片

### 5.2 资料门禁

资料未通过后台审核前：

- 不进入正式制作
- 不承诺交付时间
- 客户前台只显示 `资料需补充` 或 `已提交`

### 5.3 交付门禁

图片未通过后台复核并进入正式交付结果前：

- 不展示给客户
- 不进入下载区
- 不计为正式交付

## 6. 状态机建议

### 6.1 客户可见状态

- `待提交`
- `已提交`
- `资料需补充`
- `制作中`
- `已完成`

### 6.2 后台订单状态

- `新订单`
- `待审核`
- `需补传`
- `待制作`
- `制作中`
- `待复核`
- `待上传成果`
- `待通知客户`
- `已交付`

### 6.3 状态映射

| 后台状态 | 客户状态 |
| --- | --- |
| `新订单` / `待审核` | `已提交` |
| `需补传` | `资料需补充` |
| `待制作` / `制作中` / `待复核` | `制作中` |
| `待上传成果` / `待通知客户` | `制作中` |
| `已交付` | `已完成` |

## 7. 页面模块拆分建议

### 7.1 客户平台

客户平台建议固定为 4 个主区：

1. 样片集
2. 提交资料
3. 制作状态
4. 查看成片

每个主区只回答一个问题：

- 样片集：先选哪个模板大类、具体能做成什么样？
- 提交资料：我需要上传什么、选择什么？
- 制作状态：现在是否需要我补充？
- 查看成片：结果在哪里下载？

### 7.2 后台制作平台

后台制作平台建议固定为 5 个主区：

1. 订单列表
2. 资料审核
3. 制作记录
4. 成果上传
5. 通知与交付

后台可以展示内部细节，但必须和客户平台分开，不把制作字段直接透传到前台。

## 8. 最小可用版实施顺序

### 第一阶段

- 客户注册或登录
- 客户提交资料
- 后台管理员登录
- 后台查看订单
- 后台上传成果
- 客户查看成片

### 第二阶段

- 后台资料审核状态
- 客户补传通知
- 邮件通知
- 成果版本记录
- 下载记录

### 第三阶段

- 制作人员和复核人员角色
- 更细的后台制作状态
- 批次复核记录
- 返修记录
- 客户反馈闭环

## 9. 当前最大风险

1. 如果前台继续暴露内部生产词，客户会以为自己要操作复杂工具。
2. 如果没有客户账号，私人照片和成片无法可靠归属。
3. 如果没有管理员后台，成果上传和客户通知会散落在人工流程里。
4. 如果后台状态直接透传给客户，页面会变成管理后台。
5. 如果成果不经过复核就上传，正式交付质量不可控。

## 10. 完成标准

这份标准流程成立，至少要满足以下条件：

1. 客户平台只展示结果、资料提交、必要状态和成片。
2. 后台制作平台必须管理员登录。
3. 前台注册用户和后台管理员账号边界清楚。
4. 客户不需要理解 `三视图`、`人物确认图`、`像不像审核` 等内部动作。
5. 后台能围绕同一个定制订单审核资料、制作、上传成果和通知客户。
6. `final_4k/` 仍作为唯一正式成片目录。
7. 后续页面、状态机和实现计划都能以本文件为唯一主线入口。
