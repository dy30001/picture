# 样片管理规范

日期： 2026-05-02
模式： doc
工作区： `/Users/dy3000/code/pic`

## 0. 当前总口径

1. 样片管理是独立板块，不是临时动作
2. 样片和正式成片必须分开
3. 正式成片只认 `final_4k/`
4. `成片/` 只保留历史兼容数据，不再作为正式交付目录

## 1. 一句话结论

当前仓库不是没有样片能力，而是样片信息还分散在 4 个来源里：

1. `public/app.js` 里的样片方向预览
2. `generated/` 里的实际生成结果
3. `review/contact_sheets/` 和 `review/repair_queue_*.csv` 里的复核与修复状态
4. `final_4k/` 里的已转正成片

后续要把这 4 层收成一个正式的样片管理模块。

## 2. 样片管理放在哪个阶段

样片管理处在：

```text
三视图确认 之后
正式交付 之前
```

它负责的不是“生图本身”，而是：

- 看哪些样片已经出来
- 哪些样片被客户看过
- 哪些样片入围
- 哪些样片淘汰
- 哪些样片要重生
- 哪些样片已经转成正式交付

## 3. 当前目录与职责

| 来源 | 当前职责 |
| --- | --- |
| `public/app.js` | 样片方向预览、当前方向选择、引导后续生成 |
| `generated/` | 样片源图和中间结果 |
| `review/contact_sheets/` | 批次接触表复核 |
| `review/repair_queue_*.csv` | 待重生、修复中、待复看记录 |
| `final_4k/` | 已转正的正式成片 |
| `成片/` | 历史重复目录，不再新增正式映射 |

## 4. 样片对象

每张样片至少要有这些字段：

| 字段 | 说明 |
| --- | --- |
| `sampleId` | 样片唯一 ID |
| `projectId` | 项目或客户单号 |
| `sceneId` | 所属场景 |
| `directionId` | 所属方向 |
| `batchId` | 所属生成批次 |
| `sourcePath` | 源图或当前样片路径 |
| `thumbPath` | 缩略图路径 |
| `contactSheetPath` | 接触表路径 |
| `status` | 当前样片状态 |
| `decision` | 客户或运营决策 |
| `decisionNote` | 决策备注 |
| `repairTicket` | 对应修复记录 |
| `promotedFinalIds` | 已转正到正式成片的 ID 列表 |
| `createdAt` | 创建时间 |
| `updatedAt` | 更新时间 |

## 5. 样片状态

建议拆成两层。

### 5.1 方向层状态

- `未选择`
- `已选择`
- `样片生成中`
- `待客户确认`
- `已确认继续`
- `需重生`
- `已关闭`

### 5.2 单张样片状态

- `待查看`
- `已查看`
- `已入围`
- `已淘汰`
- `待重生`
- `重生中`
- `已修复待复看`
- `已转正`
- `已归档`

说明：

- 方向状态回答“这个方向还要不要继续”
- 单张状态回答“这一张样片现在如何处理”

## 6. 样片操作

样片管理至少要支持以下动作：

1. `设为当前方向`
2. `标记已看`
3. `入围`
4. `淘汰`
5. `标记需重生`
6. `写备注`
7. `对比修复前后`
8. `确认转正到 final_4k`
9. `撤回转正`
10. `批量导出选片结果`

这些动作不能只停留在口头流程里，必须有记录。

## 7. 样片与成片的边界

必须明确：

1. 样片不是正式交付
2. 被淘汰样片默认不进交付页
3. 被修复替换的旧样片默认归档
4. 只有 `已转正` 的样片，才进入 `final_4k/`
5. `成片/` 不再承接新的正式交付动作

也就是说：

- 样片页：负责比较、入围、淘汰、重生
- 成片页：负责复核、核数量、下载、确认交付

## 8. 界面要求

样片管理建议作为正式页面或正式大面板存在。

页面结构建议：

1. 顶部摘要
   - 当前场景
   - 当前方向
   - 待看数量
   - 入围数量
   - 待重生数量
   - 已转正数量
2. 筛选栏
   - 场景
   - 方向
   - 状态
   - 是否入围
   - 是否待重生
3. 样片列表
   - 缩略图
   - 状态
   - 风险
   - 时间
   - 当前操作
4. 详情区
   - 大图
   - 备注
   - 修复记录
   - 对应正式成片
5. 修复队列
   - 待重生
   - 修复中
   - 修复完成待复看

## 9. 当前实施建议

### 9.0 当前已落地 MVP

前端已先在 `拍摄定制 -> 样片集` 落一个轻量样片管理闭环：

- 样片集先选大场景，再看样片组矩阵，当前组内照片同屏预览
- 样片组优先按 `final_4k/` 子目录自动成组，不再把不同主题目录的照片平铺混排
- 样片墙只展示当前样片组内照片，点击照片只更新同屏预览，不打开新详情屏
- 每张样片有状态记录：`待查看 / 已入围 / 已淘汰 / 待重生 / 已转正`
- 每张样片有操作入口：`入围 / 淘汰 / 需重生 / 转正记录`
- 样片区顶部有统计摘要：`待看 / 入围 / 待重生 / 已转正`
- 状态先写入本地 `studio` 状态，不移动 `generated/`、`review/`、`final_4k/` 文件
- `已转正` 当前只是转正意向记录，正式交付仍只认 `final_4k/`

### 9.1 第一阶段

- 先统一样片对象和状态口径
- 先把 `generated/`、`review/`、`final_4k/` 的映射关系记清楚

### 9.2 第二阶段

- 把样片列表、修复队列、转正记录做成正式模块

### 9.3 第三阶段

- 再把它迁入统一代码树里的 `sample-management` 子模块

## 10. 验证方案

需要至少能回答下面 5 个问题：

1. 某张样片属于哪个场景、哪个方向、哪个批次
2. 某张样片当前状态是什么
3. 某张样片有没有修复记录
4. 某张样片是否已经转正到 `final_4k/`
5. 某张样片对应的正式交付是否还能追到来源
