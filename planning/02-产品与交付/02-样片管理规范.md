# 样片管理规范

日期： 2026-05-02
模式： doc
工作区： `/Users/dy3000/code/pic`

## 0. 当前总口径

1. `样片集` 是客户看样片的独立功能，只负责定调、选模板和看参考效果
2. 内部样片管理是后台生产和复核动作，不能混进客户看片界面
3. 样片和正式成片必须分开
4. 正式成片只认 `final_4k/`
5. `成片/` 只保留历史兼容数据，不再作为正式交付目录

## 1. 一句话结论

当前仓库不是没有样片能力，而是样片信息还分散在 4 个来源里：

1. `public/app.js` 里的样片方向预览
2. `generated/` 里的实际生成结果
3. `review/contact_sheets/` 和 `review/repair_queue_*.csv` 里的复核与修复状态
4. `final_4k/` 里的已转正成片

后续要把这 4 层收成一个正式的内部样片管理模块；客户侧仍只看到 `样片集`。

## 2. 样片管理放在哪个阶段

本文把样片拆成两层：

1. 客户可见的 `样片集`：处在提交资料之前，用于看效果、选模板大类、看子场景和参考样片。
2. 内部样片管理：处在资料审核或内部身份基准完成之后、正式交付之前，用于生产复核、入围、重生和转正。

内部样片管理负责的不是“生图本身”，而是：

- 看哪些样片已经出来
- 哪些样片被客户看过
- 哪些样片入围
- 哪些样片淘汰
- 哪些样片要重生
- 哪些样片已经转成正式交付

## 3. 当前目录与职责

| 来源 | 当前职责 |
| --- | --- |
| `public/app.js` | 客户侧样片集、模板大类、子场景、同屏大图和缩略图预览 |
| `generated/` | 样片源图和中间结果 |
| `review/contact_sheets/` | 批次接触表复核 |
| `review/repair_queue_*.csv` | 待重生、修复中、待复看记录 |
| `final_4k/` | 已转正的正式成片 |
| `成片/` | 历史重复目录，不再新增正式映射 |

## 4. 样片对象

每张样片至少要有这些字段：

| 字段 | 说明 |
| --- | --- |
| `sampleId` | 样片唯一 ID |
| `projectId` | 项目或客户单号 |
| `sceneId` | 所属场景 |
| `directionId` | 所属方向 |
| `batchId` | 所属生成批次 |
| `sourcePath` | 源图或当前样片路径 |
| `thumbPath` | 缩略图路径 |
| `contactSheetPath` | 接触表路径 |
| `status` | 当前样片状态 |
| `decision` | 客户或运营决策 |
| `decisionNote` | 决策备注 |
| `repairTicket` | 对应修复记录 |
| `promotedFinalIds` | 已转正到正式成片的 ID 列表 |
| `createdAt` | 创建时间 |
| `updatedAt` | 更新时间 |

## 5. 样片状态

建议拆成两层。

### 5.1 方向层状态

- `未选择`
- `已选择`
- `样片生成中`
- `待客户确认`
- `已确认继续`
- `需重生`
- `已关闭`

### 5.2 单张样片状态

- `待查看`
- `已查看`
- `已入围`
- `已淘汰`
- `待重生`
- `重生中`
- `已修复待复看`
- `已转正`
- `已归档`

说明：

- 方向状态回答“这个方向还要不要继续”
- 单张状态回答“这一张样片现在如何处理”

## 6. 样片操作

样片管理至少要支持以下动作：

1. `设为当前方向`
2. `标记已看`
3. `入围`
4. `淘汰`
5. `标记需重生`
6. `写备注`
7. `对比修复前后`
8. `确认转正到 final_4k`
9. `撤回转正`
10. `批量导出选片结果`

这些动作不能只停留在口头流程里，必须有记录。

## 7. 样片与成片的边界

必须明确：

1. 样片不是正式交付
2. 被淘汰样片默认不进交付页
3. 被修复替换的旧样片默认归档
4. 只有 `已转正` 的样片，才进入 `final_4k/`
5. `成片/` 不再承接新的正式交付动作

也就是说：

- 样片页：负责比较、入围、淘汰、重生
- 成片页：负责复核、核数量、下载、确认交付

## 8. 界面要求

样片管理建议作为正式页面或正式大面板存在。

页面结构建议：

1. 顶部摘要
   - 当前场景
   - 当前方向
   - 待看数量
   - 入围数量
   - 待重生数量
   - 已转正数量
2. 筛选栏
   - 场景
   - 方向
   - 状态
   - 是否入围
   - 是否待重生
3. 样片列表
   - 缩略图
   - 状态
   - 风险
   - 时间
   - 当前操作
4. 详情区
   - 大图
   - 备注
   - 修复记录
   - 对应正式成片
5. 修复队列
   - 待重生
   - 修复中
   - 修复完成待复看

## 9. 当前实施建议

### 9.0 当前已落地 MVP

前端已先在 `影像定制 -> 样片集` 落一个轻量样片管理闭环：

- 样片集先选大场景，再看样片组矩阵，当前组内照片同屏预览。
- 当前交互固定为：上方选模板大类；下方左侧列子场景/样片组；右侧默认显示当前大图，点击可放大；底部缩略图只负责切换同组照片。
- 上方模板大类必须平铺展示，不使用横向滚动条；客户一眼看到当前可选大类。
- 左侧样片组列表必须保留内部滚动位置，点击组卡或缩略图时不能回弹到顶部。
- 左侧样片组列表和右侧大图区域在桌面端必须上下齐平，形成一个完整看片工作台。
- 样片图优先服务人像识别：模板封面、样片组卡、缩略图使用接近正方形的矩形；右侧大图使用方形画框或接近方形画框，裁切优先保证脸部可见，不能做成只看到下巴以下的横条图。
- 当前组内缩略图应尽量显示完整组内照片，当前已按每组 9 张作为主要检查口径。
- 样片集只做看样片，不显示提交资料、生成三视图、选生成参数、后台制作状态等流程
- 样片组优先按 `final_4k/` 子目录自动成组，不再把不同主题目录的照片平铺混排
- 样片墙只展示当前样片组内照片，点击照片只更新同屏预览，不打开新详情屏
- `待查看 / 已入围 / 已淘汰 / 待重生 / 已转正` 是内部样片管理状态，不在客户样片集里展开
- `入围 / 淘汰 / 需重生 / 转正记录` 是后台制作或运营操作，不作为客户看片主动作
- `待看 / 入围 / 待重生 / 已转正` 这类统计摘要只进入内部管理台，不进入客户样片集首屏
- 状态先写入本地 `studio` 状态，不移动 `generated/`、`review/`、`final_4k/` 文件
- `已转正` 当前只是转正意向记录，正式交付仍只认 `final_4k/`

### 9.1 第一阶段

- 先统一样片对象和状态口径
- 先把 `generated/`、`review/`、`final_4k/` 的映射关系记清楚

### 9.2 第二阶段

- 把样片列表、修复队列、转正记录做成正式模块

### 9.3 第三阶段

- 再把它迁入统一代码树里的 `sample-management` 子模块

## 10. 验证方案

需要至少能回答下面 5 个问题：

1. 某张样片属于哪个场景、哪个方向、哪个批次
2. 某张样片当前状态是什么
3. 某张样片有没有修复记录
4. 某张样片是否已经转正到 `final_4k/`
5. 某张样片对应的正式交付是否还能追到来源
