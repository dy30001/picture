# 主应用邮箱注册说明

日期： 2026-05-02
模式： doc

## 1. 当前确认口径

- 当前合并后的主入口统一准备按 `9999` 认，不再把 `4184` 当成当前改动目标。
- `4184` 对应仓库里的 `identity-workflow/` 独立分支，只能作为历史拆分页参考，不代表当前主应用落点。
- 后续所有“注册”“邮箱验证码”“邮箱配置”“发件人名称”相关改动，都应落在当前主应用，而不是继续扩展旧独立页。

## 2. 当前主应用落点

当前主应用代码范围：

- 前端页面：`public/index.html`
- 前端逻辑：`public/app.js`
- 前端样式：`public/styles.css`
- 主服务：`server/index.mjs`

当前主应用已经具备：

- `墨境个人AI摄影棚` 主标题
- `注册开通` 一级入口
- `registerPanel` 注册说明面板
- `modalRoot` 通用弹层容器

当前主应用尚未具备：

- 站内邮箱注册
- 站内邮箱验证码发送
- 站内登录态持久化

## 3. 邮箱注册方向

当前确认的注册方向不是继续跳外链注册页，而是改成主应用内完成：

1. 邮箱验证码发送
2. 邮箱注册
3. 邮箱登录
4. 注册后保留当前主应用内的账号状态

当前 `public/app.js` 里的 `registerNowBtn` 仍然是打开外链注册页，这只是旧过渡方案，后续应替换为站内邮箱注册流程。

## 4. 邮箱配置口径

- 邮箱方案使用邮箱验证码，不走短信验证码。
- QQ 邮箱可以用 SMTP 授权码，不使用邮箱登录密码。
- 真实密钥不写入仓库，不写进 README，不写进截图。
- 发件人名称默认跟站点主题走，当前建议口径：

```text
墨境个人AI摄影棚
```

如果后续主品牌文案调整，发件人名称同步调整。

## 5. 与旧独立页的关系

- `identity-workflow/` 里的邮箱注册实现，可以作为接口、冷却时间、验证码 TTL、SMTP 发信的参考样例。
- 但当前需求不是继续把那套独立页作为主入口，而是把能力迁回主应用。
- 因此后续评审或实现时，先看主应用 `public/` + `server/index.mjs`，再决定是否复用旧代码片段。

## 6. 后续实施边界

后续如果继续做邮箱注册，优先顺序应为：

1. 主应用内注册面板改造
2. 主服务增加邮箱验证码与注册接口
3. `.env.example` 补主应用邮箱配置说明
4. 自动化测试覆盖主应用注册流

不应再做的方向：

- 不把 `4184` 当成当前唯一主入口继续堆功能
- 不把邮箱注册继续做成“先跳外链再回来填 Key”的长期方案
- 不把真实邮箱账号、授权码直接写进仓库文件
