# 香港公网更新正确方法

## 1. 结论

- 这次对的更新方法是：`代码走 Git push + 服务器 pull`，`图片走直拷`。
- 不要再把“大量模板预览图、图标和其他静态图片”跟代码一起当成一次 Git 推送的阻塞条件。
- 公网是否真的更新成功，必须同时看 `提交号`、`服务状态`、`公网 HTML`、`公网图片响应`，不能只看 `push 成功`。

## 2. 适用场景

- `代码改动为主`
  - 前端文案、交互、服务端逻辑、测试、文档、脚本。
- `图片资源改动为主`
  - 新 icon、模板预览图、封面图、样片图、分享图。
- `代码 + 图片一起改`
  - 按“两条线”处理，不要混成一步。

## 3. 正确流程

### A. 代码发布

1. 先在本地只整理代码提交。
2. 跑发布前校验：
   - `npm run lint:strict`
   - `npm test`
   - `npm run typecheck`
   - `npm run verify:visual`
3. 只把代码 push 到 GitHub。
4. 香港服务器执行：
   - `git fetch origin`
   - `git pull --ff-only origin main`
   - `systemctl restart img-inklens.service`

### B. 图片发布

1. 图片不要卡在 Git push 里等。
2. 需要上线的图片直接拷到香港服务器对应目录。
3. 典型目录：
   - `/opt/picture/public/assets/`
   - `/opt/picture/public/template-previews/`
4. 图片更新后，再验公网图片 URL 是否返回 `200`。

### C. 最终验收

1. 看服务器提交号是否到目标提交。
2. 看服务是否 `active`。
3. 看机内健康检查：
   - `curl http://127.0.0.1:9999/api/health`
4. 看公网健康检查：
   - `curl https://img.inklens.art/api/health`
5. 看公网 HTML 是否已经是新文案、新标题、新入口。
6. 看公网图片 URL 是否返回 `200`，尤其是：
   - icon
   - 当前页面首屏依赖图
   - 新增模板预览图

## 4. 这次为什么之前会错

- 之前把“代码发布”和“图片发布”混成了一步。
- 只要图片量一大，Git 推送就容易变慢、卡住或让人误以为“公网没更新”。
- 香港机即使已经 `pull` 了代码，如果图片没同步到位，公网看起来仍然像“还是原来的”。
- 还有一类误判是：只看服务健康，不看实际 HTML 和图片，容易把“服务活着”误判成“页面已更新”。

## 5. 这次的正确信号

- GitHub `main` 上先有正确代码提交。
- 香港机实际 `HEAD` 到了目标提交。
- 服务重启后 `api/health` 正常。
- 公网首页返回了新版标题和新版菜单文案。
- 公网 icon / 图片 URL 返回 `200`。

## 6. 后续固定规则

- 以后发香港公网，默认按下面执行：
  - `push 只是代码`
  - `图片直接拷贝`
  - `服务器 pull 代码`
  - `重启服务`
  - `验 HTML + 验图片 + 验健康`
- 如果图片量大，不要再把它们当成 Git 推送是否完成的判断前提。
- 如果只是代码微调，不要重复上传整包图片。
