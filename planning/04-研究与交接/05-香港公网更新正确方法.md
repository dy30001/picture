# 香港公网代理与图片同步正确方法

## 1. 结论

- 香港服务器继续保留为公网入口和备用资源节点。
- 公网 `https://img.inklens.art` 以香港 Nginx 为入口，可通过反向代理访问 Mac mini 本机 `127.0.0.1:9999`。
- 图片资源与本地保持同步，获取图片资源时可以直接从香港服务器读取，不再需要额外“备份图片”链路。
- `4184` 不作为这次公网主代理入口，不纳入这次发布链路；这次只对接 `9999`。
- 公网是否更新成功，重点看 `香港服务`、`反向隧道`、`Nginx 配置`、`公网 HTML`、`公网图片响应` 和 `图片同步状态`。

## 2. 适用场景

- `公网访问异常`
  - 先查香港 Nginx 和香港服务是否正常，再查 Mac mini 到香港的 `9999` 反向隧道。
- `本机代码改动`
  - 本机服务更新后，公网会通过反向代理访问本机 `9999`。
- `图片资源改动`
  - 本地图片仍要同步到香港资源目录，公网和协作方可直接从香港服务器取图。
- `香港服务器维护`
  - 维护 Nginx、证书、SSH 隧道入口、derp、香港服务和图片资源目录。

## 3. 正确流程

### A. 香港公网代理配置

1. 香港 `img.inklens.art` 的 Nginx 站点保持公网入口职责：
   - `/derp` -> `http://127.0.0.1:33445`
   - `/` -> 按当前配置代理到可用的主应用入口
   - 图片资源路径 -> 优先从香港本地已同步资源目录读取，必要时再走主应用代理
2. 不要改 `4184`。
3. 不要删除香港服务和图片资源目录。
4. 修改 Nginx 后必须执行：
   - `nginx -t`
   - `systemctl reload nginx`

### B. 反向隧道

1. `4184` 不作为本次公网主代理口，不要拿它承载 `img.inklens.art`。
2. 新公网主代理使用独立 LaunchAgent：
   - `~/Library/LaunchAgents/com.dy3000.img-inklens-proxy9999.plist`
3. 该隧道必须是：
   - 香港 `127.0.0.1:9999`
   - 对接 Mac mini `127.0.0.1:9999`
4. 香港侧如使用隧道入口，检查：
   - `ss -ltnp | grep :9999`
   - 正常时可看到 `sshd` 监听 `127.0.0.1:9999`。
5. Mac mini 检查：
   - `launchctl print gui/$(id -u)/com.dy3000.img-inklens-proxy9999`
   - 正常时应为 `state = running`。

### C. 图片同步

1. 本地图片资源和香港图片资源保持同步。
2. 图片同步是资源发布链路，不再叫“备份图片”。
3. 图片量大时，图片走同步/直拷，不混进 Git 代码发布。
4. 香港服务器可以作为图片资源获取源：
   - 模板预览图
   - 首页和首屏依赖图
   - `final_4k/` 正式成片
   - 其他客户可见静态素材
5. 同步后必须抽查香港图片 URL 或香港本地文件，确认关键图片可读。

### D. 最终验收

1. 香港本地服务保持可维护状态：
   - `systemctl is-active img-inklens.service`
   - `systemctl is-enabled img-inklens.service`
   - 不再把 `inactive/disabled` 作为默认正确状态。
2. 香港本地或代理健康检查：
   - `curl http://127.0.0.1:9999/api/health`
3. 公网健康检查：
   - `curl https://img.inklens.art/api/health`
4. 公网 HTML 是否来自当前主应用入口。
5. 香港图片资源是否与本地同步。
6. 公网图片 URL 是否返回 `200`，尤其是：
   - icon
   - 当前页面首屏依赖图
   - 模板预览图
   - `final_4k/` 正式成片

## 4. 这次为什么要改

- 之前把“图片同步”理解成“备份图片”，容易误删这条资源链路。
- 实际口径是：香港服务保留，图片资源与本地同步，公网取图可以直接走香港服务器。
- 代码发布和图片资源发布仍然分开：代码走 Git/服务更新，图片走同步/直拷。
- 这样既保留香港公网能力，也避免大批图片塞进代码发布链路。

## 5. 这次的正确信号

- 香港公网入口可用。
- 香港服务不要被默认停用或删除。
- 如启用 `9999` 反向隧道，香港 `127.0.0.1:9999` 可由 `sshd` 监听。
- Nginx `img.inklens.art` 的 `/` 指向当前可用主应用入口。
- `curl https://img.inklens.art/api/health` 返回 `ok:true`。
- 香港图片资源和本地图片资源一致。
- 公网首页和关键图片 URL 都返回 `200`。

## 6. 后续固定规则

- 保持香港服务器和香港服务。
- 图片资源继续与本地同步；获取图片资源可以直接走香港服务器。
- 不要把图片同步写成“备份图片”。
- 不要把大批图片混进 Git 代码发布链路。
- 不要默认停用 `img-inklens.service`，除非新的计划明确要求。
- 不要把 `4184` 接入本次公网主代理，它不是这次的目标端口。
- 如果公网异常，优先检查：
  - Mac mini `9999` 是否健康
  - `com.dy3000.img-inklens-proxy9999` 是否 running
  - 香港服务是否健康
  - 香港 `127.0.0.1:9999` 是否按当前方案监听
  - Nginx 是否指向当前主应用入口
  - 香港图片资源是否同步
