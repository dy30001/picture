# 墨境文档体系规范

日期： 2026-05-02
模式： doc
工作区： `/Users/dy3000/code/pic`

## 1. 目标

这套规范只解决 4 件事：

1. 让文档入口统一，不再分散在根目录和 `planning/`。
2. 让正式规范、执行计划、过程报告各归各位。
3. 让后来者知道什么是当前口径，什么只是历史记录。
4. 让新增文档时有固定落点和固定维护动作。

## 2. 文档分层

| 层级 | 目录 | 保存什么 | 不保存什么 |
| --- | --- | --- | --- |
| 顶层入口 | `README.md`, `NEXT_TASKS.md` | 对外说明、快速开始、当前停点 | 细碎过程记录、一次性分析 |
| 文档治理层 | `docs/` | 文档规范、统一入口、长期索引 | 每次任务的临时执行细节 |
| 文档导出层 | `docs/exports/` | HTML、PDF、封面图等导出产物 | 手工维护的源文档 |
| 工作文档层 | `planning/` | 规格、计划、交接、流程规则、执行报告 | 通用产品介绍、构建产物 |
| 内容数据层 | `public/README_zh.md` 等 | 模板库说明、数据来源说明 | 内部开发规范 |
| 产物层 | `dist/` | 打包副本 | 手工维护的源文档 |

## 3. 当前源文档口径

下面这些口径必须在文档里保持一致：

- 内部统一服务端口：`9999`
- 客户前台入口：只展示域名和品牌，不展示内部端口号
- 客户前台不展示备案号
- 默认主服务：`server/index.mjs`
- 正式成片目录：`final_4k/`
- 历史重复目录：`成片/`
- 人像基线流程：`身份确认 -> 模板样片 -> 批量场景 -> 复核 -> final_4k`

如果新文档与以上口径冲突，先修正规范和索引，再新增内容。

## 4. 命名规则

### 4.1 根目录

- `README.md`
  - 只保留对外总说明、快速开始、核心结构和统一入口。
- `NEXT_TASKS.md`
  - 只保留当前交接状态和下一步，不堆长期规则。
- 其他根目录 Markdown
  - 只有在必须作为独立交接入口时才允许新增，否则放进 `docs/` 或 `planning/`。

### 4.2 `docs/`

- 长期规范和索引使用固定编号。
- 推荐格式：`NN-主题名.md`
- 当前已保留：
  - `README.md`
  - `01-文档体系规范.md`

### 4.3 `planning/`

- `01-架构与规范/`
  - 品牌、架构、模块和统一入口规范。
- `02-产品与交付/`
  - 产品设计、客户流程、样片管理和验收规范。
- `03-流程与提示词/`
  - 身份锁定、工作流、提示词、风格矩阵和服装策略。
- `04-研究与交接/`
  - 研究、专项说明、线程交接和长期上下文。
- `05-执行计划/主题-执行计划.md`
  - 单次任务计划。
- `06-执行报告/主题-执行报告.md`
  - 执行结果和过程证据。
- `06-执行报告/主题-YYYYMMDDHHMMSS-执行报告.md`
  - 批次型、多次运行型历史记录。
- `07-结构化清单/*.csv`
  - 与计划、批次、镜头、分配相关的结构化辅助文件。

命名硬规则：

- 长期源文档统一用 `编号-中文主题.md`。
- 过程计划统一用 `主题-执行计划.md`，必要时可加补充说明，不再统一挂 `v01/v02`。
- 过程报告统一用 `主题-执行报告.md` 或 `主题-YYYYMMDDHHMMSS-执行报告.md`。
- 结构化清单统一用 `主题.csv`；确需并行多版时，用 `主题-第一版.csv`、`主题-第二版.csv`。
- 不再在源文档文件名里混用随机尾码、无意义英文缩写和通用版本壳。
- 提示词正文、命令、接口字段、代码路径可保留英文或代码态；说明性叙述、标题、索引和结论统一用中文。

## 5. 新文档落点规则

按内容类型落位：

- 对外说明、启动方式、仓库总览：`README.md`
- 长期规则、命名规范、统一入口：`docs/`
- 品牌、架构、结构规划：`planning/01-架构与规范/`
- 产品设计、客户流程、交付规范：`planning/02-产品与交付/`
- 工作流、门禁、提示词与风格策略：`planning/03-流程与提示词/`
- 研究说明和线程交接：`planning/04-研究与交接/`
- 单次任务计划：`planning/05-执行计划/主题-执行计划.md`
- 单次任务结果、证据、验收记录：`planning/06-执行报告/主题-执行报告.md`
- 批次型历史记录：`planning/06-执行报告/主题-YYYYMMDDHHMMSS-执行报告.md`
- CSV 清单和批次表：`planning/07-结构化清单/主题.csv`
- 当前线程停点：`NEXT_TASKS.md`

禁止把单次执行记录直接写进根目录长期 README。

## 6. 维护规则

每次新增或修改文档时，至少做下面这些动作：

1. 确认文档落点是否符合本规范。
2. 如果是长期入口文档，更新 `docs/README.md`。
3. 如果是 `planning/` 主线文档，更新 `planning/README.md`。
4. 如果是正式口径变化，回写 `README.md`。
5. 如果新版本替代旧版本，在新文档开头说明“谁取代了谁”。

## 7. 执行报告处理规则

`*-执行报告.md` 和 `*-YYYYMMDDHHMMSS-执行报告.md` 是历史证据，不是当前唯一口径。

使用规则：

- 可以保留执行过程、截图、命令、产物路径。
- 不能代替正式规格和统一入口。
- 如果执行报告里沉淀出长期规则，要提炼回 `docs/` 或 `planning/01-04` 的正式文档。

## 8. 交接规则

- 当前交接只认 `NEXT_TASKS.md`。
- 跨阶段背景和较长上下文放 `planning/04-研究与交接/`。
- `NEXT_TASKS.md` 只写“当前结果、停点、下一步、硬约束”，不重复整套背景。

## 9. 完成检查

文档任务完成前，至少自查这 6 项：

1. 入口有没有收敛到 `README.md -> docs/README.md -> planning/README.md`。
2. 新文档有没有放对目录。
3. 是否把过程文档和正式规范分开了。
4. 是否明确当前口径和历史记录的边界。
5. 是否同步更新了必要索引。
6. 是否避免在多个文件里重复维护同一份状态。
