# 墨境文档体系规范

Date: 2026-05-02
Mode: doc
Workspace: `/Users/dy3000/code/pic`

## 1. 目标

这套规范只解决 4 件事：

1. 让文档入口统一，不再分散在根目录和 `planning/`。
2. 让正式规范、执行计划、过程报告各归各位。
3. 让后来者知道什么是当前口径，什么只是历史记录。
4. 让新增文档时有固定落点和固定维护动作。

## 2. 文档分层

| 层级 | 目录 | 保存什么 | 不保存什么 |
| --- | --- | --- | --- |
| 顶层入口 | `README.md`, `NEXT_TASKS.md` | 对外说明、快速开始、当前停点 | 细碎过程记录、一次性分析 |
| 文档治理层 | `docs/` | 文档规范、统一入口、长期索引 | 每次任务的临时执行细节 |
| 工作文档层 | `planning/` | 规格、计划、交接、流程规则、build report | 通用产品介绍、构建产物 |
| 内容数据层 | `public/README_zh.md` 等 | 模板库说明、数据来源说明 | 内部开发规范 |
| 产物层 | `dist/` | 打包副本 | 手工维护的源文档 |

## 3. 当前源文档口径

下面这些口径必须在文档里保持一致：

- 对外统一入口：`9999`
- 默认主服务：`server/index.mjs`
- 正式成片目录：`final_4k/`
- 历史重复目录：`成片/`
- 人像基线流程：`三视图 -> 模板样片 -> 批量场景 -> review -> final_4k`

如果新文档与以上口径冲突，先修正规范和索引，再新增内容。

## 4. 命名规则

### 4.1 根目录

- `README.md`
  - 只保留对外总说明、快速开始、核心结构和统一入口。
- `NEXT_TASKS.md`
  - 只保留当前交接状态和下一步，不堆长期规则。
- 其他根目录 Markdown
  - 只有在必须作为独立交接入口时才允许新增，否则放进 `docs/` 或 `planning/`。

### 4.2 `docs/`

- 长期规范和索引使用固定编号。
- 推荐格式：`NN-主题名.md`
- 当前已保留：
  - `README.md`
  - `01-文档体系规范.md`

### 4.3 `planning/`

- `*_spec_vNN.md`
  - 长期规格文档。
- `plan_*_vNN.md`
  - 执行计划，说明要做什么。
- `build_report_*_vNN.md`
  - 执行结果和过程证据，说明做了什么。
- `*_workflow*.md` / `*_gate*.md`
  - 长期流程和门禁规则。
- `thread_handoff_YYYY-MM-DD.md`
  - 线程交接文档。
- `*.csv`
  - 与计划、批次、镜头、分配相关的结构化辅助文件。

## 5. 新文档落点规则

按内容类型落位：

- 对外说明、启动方式、仓库总览：`README.md`
- 长期规则、命名规范、统一入口：`docs/`
- 设计规格、结构规划、工作流说明：`planning/`
- 单次任务计划：`planning/plan_*`
- 单次任务结果、证据、验收记录：`planning/build_report_*`
- 当前线程停点：`NEXT_TASKS.md`

禁止把单次执行记录直接写进根目录长期 README。

## 6. 维护规则

每次新增或修改文档时，至少做下面这些动作：

1. 确认文档落点是否符合本规范。
2. 如果是长期入口文档，更新 `docs/README.md`。
3. 如果是 `planning/` 主线文档，更新 `planning/README.md`。
4. 如果是正式口径变化，回写 `README.md`。
5. 如果新版本替代旧版本，在新文档开头说明“谁取代了谁”。

## 7. build report 处理规则

`build_report_*` 是历史证据，不是当前唯一口径。

使用规则：

- 可以保留执行过程、截图、命令、产物路径。
- 不能代替正式规格和统一入口。
- 如果 build report 里沉淀出长期规则，要提炼回 `docs/` 或 `planning/*_spec*`。

## 8. 交接规则

- 当前交接只认 `NEXT_TASKS.md`。
- 跨阶段背景和较长上下文放 `planning/thread_handoff_*.md`。
- `NEXT_TASKS.md` 只写“当前结果、停点、下一步、硬约束”，不重复整套背景。

## 9. 完成检查

文档任务完成前，至少自查这 6 项：

1. 入口有没有收敛到 `README.md -> docs/README.md -> planning/README.md`。
2. 新文档有没有放对目录。
3. 是否把过程文档和正式规范分开了。
4. 是否明确当前口径和历史记录的边界。
5. 是否同步更新了必要索引。
6. 是否避免在多个文件里重复维护同一份状态。
